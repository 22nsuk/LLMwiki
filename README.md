# LLM Wiki vNext

이 패키지는 raw source를 받아 **지속형 wiki 문서**로 정리하는 구조와,
그 wiki를 유지하는 **system corpus + ops runtime**을 함께 담은 workspace다.

## Public Mirror

- canonical private vault와 별도로, code/ops-only 공개 미러를 만들 수 있다.
- 루트 `.gitignore`는 `ops.scripts.export_public_repo`와 같은 public surface policy를 따르므로, root working tree도 public-safe tracked surface로 바로 init할 수 있게 맞춘다.
- 공개 미러는 `make public-export`로 기본적으로 `/tmp/llm-wiki-public-repo/`에 생성한다.
- 새 public 파일, 새 public prefix, 공개/비공개 경계를 바꾸는 변경은 `make sync-public-policy`로 `.gitignore`의 public block을 policy source에 맞춰 갱신한다.
- 공개 대상은 주로 `ops/`, `tests/`, `tools/`, `mk/`, `.codex/agents/`, `.github/`, [ARCHITECTURE.md](./ARCHITECTURE.md), `pyproject.toml`을 포함한 루트 개발 파일이고, `raw/`, `wiki/`, `system/`, `runs/`, `external-reports/`, generated registry/report artifact는 제외한다.
- root [AGENTS.md](./AGENTS.md)는 public-safe base instruction이고, full local vault supplement는 export에서 제외되는 `AGENTS.local.md`에 둔다.
- 공개 미러는 root [LICENSE](./LICENSE)와 [THIRD_PARTY_NOTICES.md](./THIRD_PARTY_NOTICES.md)도 함께 포함해 라이선스와 attribution 계약을 유지한다.
- `make test-public`는 현재 작업 트리에서 `public` marker 테스트만 빠르게 돌리고, `make public-check`는 exported tree 안에서 static gate와 같은 `public` tier를 다시 실행해 `ops/reports/public-check-summary.json`으로 canonical 증거를 남긴다. exported tree에서 전체 테스트를 다시 돌리고 싶으면 `make public-check-all`을 사용한다.
- `.github/workflows/ci.yml`은 test tier를 `fast`, `report-contract`, `release-closeout-regression`, `artifact-finalization`, `release-sealing`, `subprocess`, `slow`, `integration`, `integration-heavy`, `public`으로 나눠 병렬 job으로 실행하고, 별도 Windows/raw-registry/supply-chain job도 유지한다. full vault가 있으면 integration tier에서 `make release-smoke`까지 함께 돈다.
- subagent runtime이 기대하는 `.codex/agents/*.toml`도 공개 미러에 함께 포함한다.
- full-vault `ops/manifest.json` 대신 exported tree에는 `PUBLIC-EXPORT-MANIFEST.json`을 남겨, public repo 기준의 file inventory만 설명한다.

공개 미러에 남겨둘 핵심 구조 설명:
- `wiki/`는 사용자/도메인/content corpus, `system/`은 maintainer/runtime/meta corpus라는 분리는 유지한다.
- `system/system-raw-registry.md`는 raw inventory summary router, `system/system-log.md`는 append-only chronology라는 운영 개념은 유지한다.
- `ops/`는 policy, eval, schema, script를 담는 control layer다.
- public mirror는 이 구조 설명과 runtime만 공개하고, 실제 corpus/router/raw inventory 본문은 private canonical vault에 둔다.
- 구조 설명의 canonical public-safe 문서는 [ARCHITECTURE.md](./ARCHITECTURE.md)다.

## 무엇이 들어 있나

- `raw/`
  - 원본 PDF와 웹 snapshot
  - binary raw는 immutable이고, markdown snapshot은 post-ingest에 metadata 보강과 capture noise 제거 수준의 minimal normalization만 허용한다.
- `wiki/`
  - 사용자/도메인/뉴스 같은 일반 content corpus
  - public mirror에서는 corpus 본문과 live inventory를 공개하지 않는다.
- `system/`
  - maintainer/runtime/self-improvement corpus
  - `system-index.md`
  - `system-raw-registry.md`
  - `system-raw-registry/`
    - `wiki.md`
    - `system.md`
  - `system-log.md`
  - source / concept / synthesis / query / lint 페이지
- `ops/`
  - 정책 파일
  - binary eval 정의
  - lint/eval/manifest/helper script
  - reusable template
  - JSON Schema
- `runs/`
  - 실제 planning / improvement run artifact

## 추천 사용 순서

1. 먼저 `make dev-install`로 로컬 개발 환경을 만든다.
   - `uv`가 있으면 이 target은 `uv venv .venv`, `uv pip install --python .venv/bin/python -r requirements-dev.txt`, `uv pip install --python .venv/bin/python -e .`를 순서대로 실행해 editable install까지 같이 고정한다.
   - `uv`가 없으면 `python3 -m venv .venv` 뒤에 `.venv/bin/python -m pip install -r requirements-dev.txt`와 `.venv/bin/python -m pip install -e .`로 자동 fallback한다.
   - 이 과정을 한 번 끝내면 이후 `make check`, `make check-strict`, `make check-all`, `make fast-smoke`, `make test-fast`, `make test`, `make test-all`, `make test-report-contract`, `make test-artifact-finalization`, `make test-release-sealing`, `make test-subprocess`, `make test-slow`, `make test-integration`, `make test-integration-heavy`, `make test-public`은 `.venv/bin/python`이 존재할 때 자동으로 그 interpreter를 사용한다.
   - 환경이 애매하면 `make bootstrap-preflight`를 먼저 실행해 repo가 선택한 `$(PYTHON)`에서 Python 3.12+와 runtime/dev dependency가 보이는지 확인하고, `ops/reports/bootstrap-preflight-report.json`에 canonical bootstrap evidence를 남긴다. 실패하면 어떤 package가 빠졌는지와 `make dev-install` 복구 경로를 출력한다.
   - `make`를 바로 쓸 수 없는 환경이라면 같은 표준 경로로 `.venv`를 만든 뒤 `uv sync --extra dev` 또는 `.venv/bin/python -m pip install -r requirements-dev.txt && .venv/bin/python -m pip install -e .`를 사용한다.
   - 루트 `pyproject.toml`은 `ops` runtime의 build metadata를 고정하므로, packaging sanity가 필요하면 `python -m build`를 추가로 실행하면 된다.
   - 별도 virtualenv를 만들지 않고 즉시 점검만 하고 싶다면 `uv run --with pytest --with pytest-xdist --with ruff --with mypy --with types-PyYAML --with jsonschema --with pyyaml make check` 같은 ephemeral 실행도 가능하다.
   - runtime dependency만 따로 설치하고 싶다면, 이미 활성화된 가상환경 안에서 `python -m pip install -r requirements.txt`를 사용한다.
2. 일반 content나 뉴스 질문이면 `wiki/index.md`부터 읽는다.
3. 시스템 구조나 maintainer/runtime 질문이면 `system/system-index.md`부터 읽는다.
4. raw 등록 상태와 provenance가 필요하면 `system/system-raw-registry.md`를 먼저 보고, corpus별 detail이 필요하면 shard page로 내려간다.
5. 기본 점검은 `make check`로 실행한다. 이 명령은 static gate, registry preflight, lint pass, Stage 1 eval 만점, Stage 2 eval 만점, planning gate pass, 그리고 `fast` tier `pytest` pass를 함께 요구한다. full-vault closeout의 기본 명령은 `make check-finalized`다. 이 target은 먼저 auto-improve readiness ledger를 최신 routing/outcome evidence에 맞춘 뒤 `make check`를 실행하고, 이어 post-check finalizer dry-run, fixed-point finalizer, final external report action matrix refresh, strict finalizer dry-run, finality verify, 그리고 finality-attested auto-improve readiness/action matrix late refresh까지 이어서, canonical refresh를 따로 기억해야 하는 drift를 줄인다.
   - `make static`은 `ops/scripts`, `tests`, `tools`에 대한 ruff hard gate와 `MYPY_TARGETS` allowlist에 대한 mypy hard gate를 함께 실행한다.
   - `make fast-smoke`는 Subagent/developer precheck 전용 curated pytest slice다. strict import fallback/module-surface contract, report schema sample fixtures, artifact I/O helper, mechanism review runtime, mutation proposal primary-evidence fail-fast edges, auto-improve readiness runtime, artifact freshness, release-smoke unit tests만 실행하고 full wiki lint/eval, `integration-heavy` generator smoke, full `make release-smoke` package gate는 포함하지 않는다.
   - 수동 full-vault 작업에서 `system/system-log.md`를 남겨야 한다면, 로그와 generated artifact까지 먼저 갱신한 뒤 최종 tree에서 `make check-finalized` 또는 필요한 lint/eval gate를 한 번 실행한다. 작업 전 lint/eval은 baseline 진단으로만 보고, closeout 판정으로 재사용하지 않는다.
   - 개발 중 기본 테스트 루프는 `make test-fast`다. 기존 `make test`와 `make unit-tests`는 compatibility alias로 같은 fast lane을 실행한다. 이 경로는 `fast` tier만 `-p xdist.plugin -n auto --dist=loadfile`로 실행해 class/file 단위 setup cache locality를 보존하면서 wall-clock을 줄인다.
   - report schema contract는 `report_contract` lane의 core/extended pack에서 관리하고, current repo에서 실제 generator가 끝까지 도는 smoke는 `tests/test_report_generation_smoke.py`의 `integration-heavy + report_contract` 조합으로 분리한다.
   - structural complexity preview는 critical orchestrator profile과 `high_complexity_helpers` profile을 함께 보고, `diagnostics.function_budget_top_n`으로 monitored/unmonitored 함수 예산 후보 상위 N개를 드러낸다. `diagnostics.function_budget_monitoring`은 unmonitored 후보가 남아 있으면 `gate_effect: preview` / `status: attention`으로 별도 governance signal을 남긴다.
   - `make complexity-budget`은 기본 monitored profile 전체를 `ops/reports/structural-complexity-budget.json`에 쓰는 preview이고, `make complexity-budget-touched-check`는 명시 target/changed-files subset만 `ops/reports/structural-complexity-budget-touched.json`에 쓰는 strict touched-surface gate다.
   - runtime/report generator가 `generated_at`나 날짜 기반 판정을 내면 `RuntimeContext`를 받아 injected clock으로 재현 가능해야 하며, 해당 경로에는 frozen-clock 회귀 테스트를 둔다.
   - stricter readiness가 필요하면 `make warning-budget` 또는 `make check-strict`를 사용한다. 이 gate는 normal `warn`을 모두 실패로 바꾸지는 않고, policy의 `strict_warning_budget` 프로파일이 지정한 warning type 예산만 별도로 강제한다.
   - `tests/test_run_mechanism_experiment_contracts.py`는 wrapper preflight/scaffold contract를 fast tier로 두고, `tests/test_run_mechanism_experiment.py`는 실제 promote/hold/discard flow만 integration tier에 남긴다.
   - persistent pytest lane marker는 `slow`, `integration`, `integration_heavy`, `public`, `report_contract`, `artifact_finalization`, `release_sealing`, `subprocess`이고, 조합 허용/금지는 `ops/test-lane-registry.json`이 단일 권위로 관리한다. report contract는 `make test-report-contract`, checked-in generated artifact final self-check는 `make test-artifact-finalization`, release sealing regression은 `make test-release-sealing`, real subprocess/startup/timeout surface는 `make test-subprocess`로 나눠서 돌린다. 느린 deterministic 회귀는 `make test-slow`, end-to-end/live-runtime 검증은 `make test-integration`, repo-root live generator smoke는 `make test-integration-heavy`, root-tree public contract는 `make test-public`, exported-tree public contract는 static gate를 포함한 `make public-check`로 나눠서 돌린다.
   - 개발 중 전체 회귀가 필요하면 `make test-all` 또는 `make check-all`을 사용한다. 이 경로는 generated artifact currentness 수렴을 사람이 맞추지 않아도 되도록 `artifact_finalization` lane을 제외한다.
   - checked-in generated artifact final self-check는 report/generated artifact를 갱신한 뒤 `make test-artifact-finalization`, `make report-contract-closeout`, 또는 release closeout 경로에서 별도로 실행한다.
   - 병렬 실행이 환경과 충돌할 때만 `make test-serial`, `make test-all-serial`, `make check-serial` 또는 `.venv/bin/python -m pytest -m "not artifact_finalization"`로 직렬 fallback을 사용한다.
   - 명시적으로 병렬 개발 full suite를 호출하고 싶다면 `make test-all-parallel` 또는 `.venv/bin/python -m pytest -m "not artifact_finalization" -p xdist.plugin -n auto --dist=loadfile`를 사용하면 된다.
   - tiered test file은 각 module의 `pytestmark`로 registry-defined lane marker를 직접 선언한다. `integration_heavy`는 plain `integration`에 자동 포함되지 않고, `report_contract`/`public`/`release_sealing` 같은 multi-axis 조합 허용 여부도 registry가 막는다.
   - `pytest.ini`는 discovery 범위와 `--capture=sys`를 고정해, 지원 Python 3.12+ 환경에서 별도 옵션 없이 같은 진입점을 재현하게 한다.
   - 공식 pytest 진입점은 `make test*`, `make check*`, `make public-check*` 또는 `.venv/bin/python -m pytest`다. `pytest.ini`에 `pythonpath = .`를 넣지 않으므로 plain `pytest`는 지원 계약이 아니며, `tests/conftest.py`가 지원 진입점으로 안내하는 guided failure를 낸다.
   - 문서, CI, 재현 절차 예시도 bare `pytest`가 아니라 `python -m pytest`(또는 repo venv interpreter의 `-m pytest`)를 사용한다. plain `pytest` 실패는 repo 버그보다 caller-contract mismatch로 분류한다.
   - 공식 `make test*`, `make check*`, `make public-check*` 경로는 `PYTEST_DISABLE_PLUGIN_AUTOLOAD=1`을 export해 외부 pytest plugin drift를 차단한다. 직접 `.venv/bin/python -m pytest`를 호출할 때도 같은 환경 변수를 맞추는 편이 좋다.
   - `ruff` 규칙군 확대 실험은 기본 gate와 분리된 `make ruff-strict-preview`로 돌린다. 이 타깃은 `ops/ruff-strict-preview-allowlist.txt`에 적힌 파일/폴더만 읽어 `B`, `SIM`, `UP`, `I` 규칙을 opt-in으로 적용하므로, helper surface를 단계적으로 넓힐 때 allowlist만 갱신하면 된다.
   - `mypy` 강화도 전체 일괄 전환 대신 `make mypy-strict-preview`로 먼저 본다. 이 타깃은 `ops/mypy-strict-preview-allowlist.txt`에 적힌 최근 변경 핵심 runtime helper에만 `--check-untyped-defs`, `--disallow-untyped-defs`, `--disallow-incomplete-defs`를 추가 적용한다.
6. 실제 배포 전에는 먼저 `make release-ready`를 실행해 공개 미러까지 포함한 canonical release evidence 수렴, release-ready commit, clean worktree `release-check-all-surfaces`를 한 흐름으로 닫는다. 수렴만 따로 확인해야 하면 `make release-converge` 또는 `make release-converge-all-surfaces`를 사용하고, 커밋 후 clean worktree에서 `make release-check` 또는 `make release-check-all-surfaces`를 실행한다.
   - `make release-smoke-fast`는 archive build/extract/manifest comparison과 filename budget만 빠르게 확인하고, extracted archive 안에서 `raw_registry_preflight`/`wiki_lint`/`wiki_eval`/`wiki_stage2_eval`/`planning_gate_validate`는 생략한다. 이 경로는 developer/package precheck이며 canonical release evidence로 쓰지 않는다.
   - `make release-smoke`와 alias `make release-smoke-full`은 full profile로 extracted archive 안에서 위 5개 smoke command까지 모두 다시 실행한다. `make release-smoke-full-reuse`는 현재 source tree, schema/input fingerprints, profile, passing status가 이미 닫힌 `ops/reports/release-smoke-report.json`과 같을 때 archive build/extract와 smoke command replay를 건너뛰고, 아니면 full smoke를 실행한다. `make release-check`와 release closeout이 읽는 canonical release evidence는 이 full 단일 report인 `ops/reports/release-smoke-report.json`이다.
   - `make release-converge`는 public report schema sample fixture refresh, smoke 없는 `test-execution-summary-report-contract-refresh-no-smoke`, registry preflight, reusable full release smoke, `make generated-artifact-converge`, remediation backlog, release closeout fixed-point, strict post-check finalizer, `test-artifact-finalization`을 실행해 checked-in canonical evidence를 갱신하는 mutating 단계다. `make release-converge-all-surfaces`는 여기에 `sync-public-policy`와 `public-check-all`까지 포함한다.
   - `make release-check`는 check-only gate다. 시작과 끝에서 `release-worktree-clean-check`를 실행하고, report-contract pytest를 직접 돌리며, registry/public policy/public check/release smoke는 tmp check 또는 current checked-in evidence reuse check로만 검증한다. 따라서 check가 통과하려면 canonical artifact가 이미 converge 단계에서 갱신 및 커밋되어 있어야 한다.
   - `make release-check-all-surfaces`는 같은 check-only `release-check-core` 뒤에 `sync-public-policy-check`와 `public-check-all-check`를 실행하고, 마지막에 `test-artifact-finalization` 및 clean worktree check를 다시 실행한다. 이전의 `release-check-preflight-converge`와 `release-check-post-converge`는 mutating compatibility alias로만 남는다.
   - `make release-ready`는 `release-ready-snapshot -> release-converge-all-surfaces -> release-ready-commit -> release-ready-post-commit-converge -> release-ready-amend -> release-check-all-surfaces` 순서로 실행된다. 첫 commit은 public source surface(`ops/`, `tests/`, `tools/`, `mk/`, `.codex/agents/`, `.github/`, 루트 문서/설정)와 canonical generated evidence를 같은 커밋에 넣어 clean 기준점을 만들고, post-commit converge는 clean worktree guard를 다시 snapshot한 뒤 `generated-artifact-converge -> remediation-backlog -> release-closeout-fixed-point -> strict post-check finalizer`로 readiness/backlog/closeout fingerprint를 같은 fixed-point writer set 안에서 수렴시킨다. fixed-point는 downstream closeout writers 뒤에 generated index/freshness feedback refresh를 반복해 finalization lane이 다시 stale해지지 않게 한다. 이후 `release-ready-amend`가 post-commit evidence를 첫 commit에 접어 최종 history는 release-ready commit 1개만 남기므로, self-dirty 또는 stale closeout evidence가 checked-in 증거로 남지 않는다. `raw/`, `wiki/`, `system/`, `runs/`, `external-reports/`, `tmp/` 같은 private/transient dirty path나 이미 staged된 변경이 있으면 자동 커밋하지 않고 실패한다.
   - pack/unpack 이후 manifest inventory가 source vault와 byte-level로 같은지도 함께 검사한다.
   - `make release-evidence-closeout`은 final authoritative clean lane이다. recipe 안에서 generated core/static/full smoke/source-package clean extract/full test summary/generated artifact convergence/closeout summary를 닫고, outcome/provenance refresh 뒤 auto-improve readiness를 다시 동기화한 다음 `release-closeout-post-check-finalizer-dry-run -> release-closeout-fixed-point -> tmp-json-clean -> release-closeout-finality-verify`를 수행한다. `release-closeout-fixed-point`는 auto-improve readiness, release closeout summary, signoff revalidation, release evidence cohort, dashboard/lane/blocker ledger, batch manifest, self-check를 같은 bounded writer set으로 수렴시켜 closeout이 최신 readiness fingerprint를 읽도록 한다. remediation backlog는 readiness 입력이므로 fixed-point 직전 pre-refresh로 갱신한다. planner/order-guard truth는 checked-in advisory JSON이 아니라 source-derived workflow graph와 guard runtime이 담당하고, advisory output은 `tmp/workflow-dependency-planner.json`, `tmp/release-workflow-order-guard.json`, `tmp/make-target-inventory.json`, `tmp/archive-execution-manifest.json` 같은 tmp surface로 demote됐다. batch manifest는 release evidence inventory와 release authority snapshot만 담당하고, fixed-point report는 writer set 수렴만 증명한다. 최종 digest binding 권위는 `ops/reports/release-closeout-finality-attestation.json`이며 fixed-point report, batch manifest, self-check, external report manifest, current tracked digest map을 non-cyclic seal로 묶는다. 기본 closeout은 distribution ZIP이 없으면 `sealed_release_status=unsealed_distribution_not_provided`를 유지하고, non-sealed external strict evidence는 `external_report_strict_unavailable` typed attention으로 남긴다. `make release-evidence-closeout-sealed`는 `build/release/LLMwiki-source.zip`을 materialize한 뒤 이를 `RELEASE_CLOSEOUT_DISTRIBUTION_ZIP` 및 ZIP improvement로 proposal budget이 소진된 뒤에는 단순 heartbeat padding으로 대기하지 않고, `GOAL_MAINTAIN_UNTIL_BUDGET=1`과 `GOAL_MAINTENANCE_INTERVAL_SECONDS=300`에 따라 남은 wall-clock budget 동안 mechanism review, mutation proposal, readiness, session report를 반복 갱신한다. `DISCARD`는 successful improvement가 아니라 blocking outcome으로 기록하고 failure/remediation 경로로 멈춘다. `make goal-runtime-reconcile`은 run-local `runs/goal-$(GOAL_RUN_ID)/state/` 아래에서 readiness, session synopsis, self-improvement negative lessons, remediation backlog를 fixed point까지 먼저 수렴시킨 뒤, 통과한 run-local evidence를 `ops/reports/` canonical snapshot으로 한 번만 publish하고 fixed-point check, `make generated-artifact-converge`를 이어 실행한다. `make goal-runtime-clean-transient`는 current status가 보호하는 active `runs/goal-*` state와 JSON prompt snapshot을 남기고 obsolete tracked goal markdown/resume/audit surface, stale legacy runtime event logs, tmp extraction trees를 제거한다. `make goal-runtime-run-admission-converge`는 새 run 전에 transient cleanup, quarantine/history cleanup preflight, queue input refresh, run-local evidence convergence, canonical publish, fixed-point check를 수행하는 mutating 준비 target이다. 이 target이 tracked canonical report를 갱신하면 해당 변경을 커밋하거나 명시 정리한 뒤 `make goal-runtime-run-admission`을 다시 실행한다. `make goal-runtime-run-admission`은 tracked canonical snapshot을 다시 더럽히지 않고 `goal-runtime-run-admission-local-refresh`로 transient cleanup/quarantine preflight/run-local evidence fixed point를 먼저 갱신한 뒤 최신 run-local readiness/backlog를 읽는 start gate이며, `tmp/goal-runtime-run-admission.json`에 `can_start_goal_runtime`, `can_mutate_candidate`, `can_promote_result_later`, `recommended_next_action`을 남긴다. `make auto-improve-goal-run`과 `make auto-improve-goal-resume`은 이 admission gate를 기본 의존성으로 통과해야 하며, dirty/stale worktree, unresolved archive/quarantine cleanup, fixed-point drift, runnable proposal 0개 상태에서는 runner를 시작하지 않는다. promotion-only blocker는 run 시작 가능성과 분리해 report에 attention으로 남기지만, 실제 promotion gate는 약화하지 않는다. `make long-run-preflight-clean`은 mutating converge target의 호환 alias다. active lock은 `make goal-runtime-lock-status`로 확인하고, 운영자가 중단 판단을 내린 뒤 `make goal-runtime-lock-stop`으로 SIGTERM을 요청한다.
   - Codex goal prompt는 승격 게이트를 점수표처럼 최적화하라고 주지 않고, required evidence와 현재 promotion blocker를 수리 대상으로 삼는 작업 지침으로 전달한다. 즉 readiness/source-package/public-check/release closeout/goal status/session/remediation evidence를 current하고 blocker-free하게 만들되, threshold 완화, guard 삭제, output-only report 직접 조작, risk relabeling으로 `can_promote_result=true`를 만드는 방식은 금지한다.
   - 자가개선 루프에서 code/report-generator 변경 직후 closeout은 `테스트 실패 -> 수정 -> 전체 재실행`을 기본 반복으로 삼지 않는다. 기본 운영 경로는 `make goal-runtime-closeout`이며, 이 target은 fingerprint 기반 cheap closeout plan을 사용해 먼저 `goal-runtime-closeout-candidate-converge`를 실행하고 expensive evidence가 이미 current일 때만 `goal-runtime-closeout-publish -> goal-runtime-closeout-finalize` 트랜잭션으로 이어진다. candidate 단계는 `report-schema-samples-check`, transient cleanup, run-local goal/readiness/session/backlog fixed point, run-local closeout candidate generated artifact convergence를 먼저 맞추고, publish 단계는 candidate output, goal evidence, certificate, `make generated-artifact-converge`를 단일 canonical publish boundary에서 닫으며, finalize 단계는 artifact finalization과 fixed-point check만 실행한다. full-suite/release-smoke 같은 expensive evidence는 cheap budget에서 자동 실행하지 않고, plan이 stale expensive evidence를 표시하면 canonical publish/finalization을 건너뛰고 candidate plan만 남긴다. source fingerprint가 바뀐 경우에만 운영자가 `make goal-runtime-closeout-full`을 명시해 full-suite evidence를 source fingerprint당 최대 1회 갱신한다. 수동 경로에서도 먼저 `make report-schema-samples-check`, candidate generated artifact convergence, release source-tree evidence가 바뀌었거나 불확실할 때의 `make release-smoke-full-reuse`, 마지막 `make test-artifact-finalization` 순서로 확인한다. 이 순서는 Codex goal prompt에도 기록되어 run/resume 지시로 전달된다.
   - goal run 관측성은 `ops/schemas/goal-run-status.schema.json`과 `ops/scripts/mechanism/goal_run_status.py`가 담당한다. active run에서는 `runs/goal-<id>/state/goal-run-status.json`, `status.md`, `audit-log.jsonl`, `resume-metadata.json`, `checkpoint-command-events.jsonl`이 current run evidence이고, `ops/reports/goal-run-status.json`은 명시 publish 뒤의 tracked canonical snapshot이다. status report는 goal contract digest, heartbeat/checkpoint/backoff/resume health, status markdown, audit log, resume metadata path를 함께 노출해 장기 런 중 “조용히 진행 중”과 “promotion 금지 상태”를 분리한다. `runtime_certificate`는 단일 self-improvement loop의 completed run, run artifact freshness, required evidence, command observability, session maintenance, promotion guard를 한 구조로 드러내며, 파일 존재나 elapsed time만으로 certificate verification을 주장하지 않는다. due periodic checkpoint evidence는 report `generated_at`이 checkpoint due time 이후인 fresh report이고, runner가 run-local `checkpoint-command-events.jsonl`에 성공한 command bundle을 남겼을 때만 observed로 인정한다. `make goal-runtime-certificate`는 completed run의 실제 elapsed seconds가 runtime 상한을 넘지 않았는지, final run artifacts, required full-gate evidence, current heartbeat/checkpoint/command-heartbeat 상태, command audit, repeated runtime maintenance evidence를 검사해 `ops/reports/goal-runtime-certificate.json`을 쓰고, `GOAL_RUNTIME_CERTIFICATE_APPLY=1`일 때만 process-persistent contract의 certificate 상태를 verified로 반영한다. 기존 canonical certificate가 이미 verified/pass이면 새 기본 run이 아직 blocked/not-certifiable이어도 verified certificate를 새 attention 진단으로 덮지 않는다. 새 certificate가 실제로 verified/pass가 될 때만 canonical certificate를 supersede한다. certificate report의 `diagnosis`는 status=attention이 report failure가 아니라 not-yet-certifiable 진단임을 명시하고, PROMOTE mechanism run은 matching session의 successful iteration 증거가 될 수 있지만 completed goal-run status와 runner command audit을 대체하지 못한다고 기록한다. heartbeat/backoff clock math, periodic evidence scheduling, runtime certificate evidence, resume metadata, command-runner heartbeat emission은 `goal_runtime_backoff.py`, `goal_runtime_maintenance.py`, `goal_runtime_certificate.py`, `goal_runtime_resume.py`, `goal_runtime_runner.py`로 분리되어 status writer의 장기 런 maintenance surface가 작게 유지된다. `ops/scripts/core/command_runtime.py`는 optional process heartbeat callback과 executor-report observability fields를 제공하며, stdout/stderr streaming timestamp 정밀화는 별도 구조 변경으로 남긴다.
   - `make self-improvement-negative-lessons`는 learning claim activation의 embedded negative ledger와 session synopsis forbidden-repeat pattern을 standalone `ops/reports/self-improvement-negative-lessons.json`으로 승격한다. session synopsis는 `goal-run-status`의 `runtime_certificate`를 concise `goal_runtime_certificate` handoff로 다시 노출해 다음 세션의 target runtime, missing evidence, sustained-claim ban을 먼저 보게 한다. `make remediation-backlog`는 반복 negative lesson과 active blocker를 `ops/reports/remediation-backlog.json` backlog item으로 남겨 같은 blocker를 2회 이상 반복한 뒤 곧바로 재시도하지 않게 한다. 만료 전 learning readiness signoff가 있으면 `learning_blocked_by_review_required` backlog item은 `deferred`로 전환되고, verified/pass canonical goal-runtime certificate는 certificate-incomplete item을 `closed`로 전환한다. `ops/policies/remediation-backlog-status-overrides.json`은 수리 증거가 있는 item을 `closed` 또는 `deferred`로 표시하는 durable input이며, auto-improve session runtime은 open `blocks_repeat` backlog item을 실행 전 stop으로 읽어 repair target closure를 먼저 요구한다.
   - CI의 focused release-closeout regression slice를 로컬에서 재현하려면 `make release-closeout-regression-dry-run`을 사용한다. 이 target은 focused pytest pack, tmp artifact-freshness check, finality verify를 canonical mutation 없이 묶는다.

   - 결과는 `ops/reports/mutation-proposals.json`에 저장되며, 한 proposal은 항상 `single_mechanism_scope` 하나만 제안한다.
   - `repeated_discard_runs`처럼 단일 failure_mode로 좁혀진 proposal은 candidate의 전체 eval-stagnation window가 아니라 matching `signal_run_ids` slice를 `run_ids`로 사용한다.
15. proposal 하나를 실제 run starter로 얼리고 싶을 때는 `.venv/bin/python -m ops.scripts.run_mechanism_experiment --vault . --run-id <run-id> --proposal-id <proposal-id> --scaffold-only`를 사용한다.
   - WSL에서 수동 mechanism run을 시작할 때는 Windows temp 경로를 물려받지 않도록 `make run-mechanism-experiment-linux-tmp MECHANISM_RUN_ARGS='--run-id <run-id> --proposal-id <proposal-id> --scaffold-only'`를 사용한다. 이 target은 `TMPDIR=/tmp TEMP=/tmp TMP=/tmp`를 고정한 뒤 runner를 호출한다.
   - 이 모드는 아직 실험을 실행하지 않고 `runs/<run-id>/proposal-snapshot.json`, `seed.yaml`, `plan.md`, `open-questions.md`, `improvement-observations.json`을 proposal-aware starter로 생성한 뒤 멈춘다.
   - full run의 `--mutation-command`, `--check-command`는 shell이 아니라 argv-style command로 실행된다. `&&`, `|`, redirection 같은 shell operator는 지원하지 않으므로, 복합 로직이 필요하면 별도 스크립트에 넣고 그 스크립트를 직접 호출한다.
   - wrapper는 baseline/candidate capture를 시작하기 전에 두 command를 먼저 parse/validate하므로, 잘못된 shell control syntax는 run scaffold를 만들기 전에 즉시 실패한다.
   - wrapper가 `HOLD`를 반환하면 candidate artifact, changed-files manifest, behavior-delta artifact는 run-local로 남기되, live repo에는 apply하지 않고 finalization도 건너뛴다.
   - run을 닫기 전에 작업 중 발견한 reusable automation 또는 repo hygiene follow-up이 있으면 `runs/<run-id>/improvement-observations.json`에 남겨 다음 자동화 입력으로 재사용한다.
16. run과 무관한 standalone repo maintenance 작업에서도 같은 observation artifact를 남길 수 있다.
   - `.venv/bin/python -m ops.scripts.improvement_observations --vault . --task-id <task-id>`
   - 기본 경로는 `ops/reports/task-improvement-observations/<task-id>/improvement-observations.json`이고, task closeout 전에 여기에 reusable follow-up을 남긴다.
   - 이미 backlog에 있는 follow-up을 이번 작업에서 해결했다면 새 record를 중복 생성하기보다 기존 observation의 `status`와 `suggested_followup`를 갱신한다.
17. `system_mechanism` 자가개선을 첫 live run 기준으로 bounded하게 실행하려면 `make auto-improve-goal-run PYTHON=.venv/bin/python VAULT=. GOAL_ALLOW_LEARNING_UNCERTAIN=1`를 사용한다.
   - 이 오케스트레이터는 `mechanism_review -> mutation_proposal -> scope freeze -> subagent routing -> executor -> repo health -> promotion -> finalize -> queue refresh`를 한 세션 안에서 반복한다.
   - 자동 루프는 `system_mechanism`만 다루고, 실제 repo mutation 허용 범위도 `ops/**`, `tests/**`, run artifact, append-only `system/system-log.md`로 제한된다.
   - 사람 signoff는 이 자동 경로에서 기본 요구사항이 아니며, 대신 `make check` green 유지, bounded target enforcement, atomic apply, session quarantine를 강제한다.
   - canonical policy 기본값은 이제 `auto_improve_policy.apply_mode = live`다. `PROMOTE`가 나오면 기본 경로에서 live apply와 finalize까지 진행하고, canary는 `run_mechanism_experiment --apply-mode canary_only`로만 명시적으로 선택한다.
  - 첫 live run 전에는 `make auto-improve-readiness`, `make check-serial`, `make public-check-serial`을 canonical preflight로 사용한다. `make auto-improve-readiness`는 내부에서 `make refresh-generated-core`와 Git/ZIP worktree guard를 먼저 실행하고, 최신 `outcome-metrics.json`, `mechanism-review-candidates.json`, `mutation-proposals.json`을 읽어 emitted proposal 수뿐 아니라 실제 runnable proposal queue, `can_execute_trial`, `can_promote_result`, learning/promotion blocker, fallback next step을 `ops/reports/auto-improve-readiness.json`으로 고정한다.
   - readiness report의 canonical contract는 `execution_readiness`, `learning_readiness`, `can_execute_trial`, `can_promote_result`, `learning_claim_blockers[]`, `promotion_blockers[]`, `clean_release_blockers[]`다. CLI exit code와 preflight gate는 top-level `status`가 아니라 `execution_readiness.can_run`을 기준으로 판정한다. 단, `learning_readiness.likely_to_learn`이 false이면 `learning_claim_blockers[]`에 `learning_blocked_by_*` named blocker가 남고, 실행 불가나 승격 불가 사유는 `promotion_blockers[]`에 모인다. `clean_release_blockers[]`는 source/package clean lane을 실제로 막는 readiness-origin blocker만 담으므로 learning claim review blocker와 result promotion blocker를 release blocker처럼 오해하지 않게 한다. top-level `next_action`도 execution-only 낙관 요약이 아니라 execution/learning 둘을 함께 반영하며, trial은 가능하지만 promotion이 막힌 경우에는 `Trial only; do not promote.` 문구로 승격 금지를 먼저 드러낸다. queue layer의 `queue.ready`는 runnable proposal queue snapshot용 boolean으로만 남고, legacy `combined_state`, `diagnostics.learnability_shadow_gate`, compatibility `release_blockers[]`는 새 report에서 생성하지 않는다.
  - release closeout의 learning readiness signoff는 blanket waiver가 아니다. `ops/reports/learning-readiness-signoff.json`은 `linked_blocker_id=learning_blocked_by_review_required`일 때만 효력이 있고, `learning_blocked_by_execution_not_runnable`에는 효력이 없다. not-runnable execution blocker는 source clean lane이 아니라 live auto-improve lane과 learning claim lane에 남는다. 만료 시각 이후에는 closeout이 다시 blocker를 유지한다. 만료 전에는 `make learning-readiness-signoff-revalidation`으로 `learning_readiness.likely_to_learn` 개선 여부와 signoff 갱신 필요성을 별도 리포트로 남긴다.
  - `mutation-proposals.json` selection은 unblocked proposal을 report slot에 먼저 배치한다. `recent_log_overlap` 같은 blocked proposal은 실행 큐에서 제외하되, `diagnostics.queue_selection.blocked_reason_counts`에 남겨 queue pressure를 계속 볼 수 있게 한다. 이 chronology guard는 `mutation_proposal.dedupe_window`와 `mutation_proposal.recent_log_overlap_max_age_days`를 함께 적용해, 최근 N개 log entry 안의 실제 중복은 막지만 오래된 low-activity chronology가 장기 런 queue를 영구 차단하지 않게 한다.
  - `make auto-improve-readiness`가 WARN으로 끝나면 auto-improve를 시작하지 않는다. queue가 비어 있거나 emitted proposal이 모두 `blocked_by` 상태일 때도 pause가 맞다. fallback family seed run이 없을 때는 `ops/scripts/mechanism/auto_improve_iteration_persistence_runtime.py`와 `ops/schemas/run-telemetry.schema.json`을 대상으로 `tests/test_auto_improve_iteration_runtime.py`를 붙인 수동 `system_mechanism` run을 한 번 finalized한 뒤 다시 `make auto-improve-readiness`를 실행해 queue를 재판정한다.
   - helper를 다시 실행한 뒤에도 proposal queue가 비어 있으면 auto-improve는 보류하고, candidate threshold/history 문제를 별도 maintenance 작업으로 넘긴다.
   - 주요 산출물은 `ops/reports/auto-improve-sessions/<session-id>.json`, `ops/reports/routing-provenance-aggregates/<session-id>.json`, `ops/reports/promotion-decision-trends.json`, `ops/reports/outcome-metrics.json`, `runs/<run-id>/scope-freeze.json`, `runs/<run-id>/subagent-routing.<role>.json`, `runs/<run-id>/<role>-executor-report.json`, `runs/<run-id>/behavior-delta.json`, `runs/<run-id>/run-telemetry.json`, `runs/<run-id>/run-artifact-fingerprint.json`이다.
   - `scope-freeze.json`은 focused test와 risk flag뿐 아니라 `auto_improve_policy.allowed_apply_roots`도 함께 freeze해서, promotion gate와 실제 apply 경로가 같은 write allowlist를 읽도록 만든다.
   - `behavior-delta.json`은 changed-files manifest를 baseline/candidate workspace에 대조해 surface class, Python symbol delta, declared-scope intent, contract touch, coverage gap risk를 deterministic하게 기록하는 schema-backed artifact다. 수동 non-equal `system_mechanism` 경로에서는 누락 시 WARN이지만, auto-improve 경로와 same-eval promotion에서는 promotion hard gate 입력이다.
   - `run-telemetry.json`은 이제 `ops/schemas/run-telemetry.schema.json`으로 검증되는 schema-backed artifact다.
   - `run-artifact-fingerprint.json`은 run-local artifact의 path, schema, size, sha256을 고정해 replay/parity 확인의 출발점으로 쓴다.
   - 이 fingerprint의 top-level `repo_provenance_snapshot`은 run 시작 시점의 `ops/reports/supply-chain-provenance.json` 경로, sha256, generated_at, status를 함께 고정한다. repo-level provenance report 본문을 run directory로 중복 복사하지 않고, hash snapshot 링크만 남기는 것이 기본 계약이다.
   - `routing-provenance-aggregates/<session-id>.json`은 session report, run telemetry, routing report, executor report, artifact fingerprint를 한 번에 엮고, routing/executor/telemetry `audit_rollup`을 함께 담는 schema-backed audit entrypoint다.
   - session report는 운영 상태의 canonical source이고, routing provenance aggregate는 그 값을 감사자가 빠르게 읽을 수 있게 고정한 normalized snapshot이다.
   - `promotion-decision-trends.json`은 `runs/*/promotion-report.json`을 훑어 decision/artifact class trend와 최근 promotion window를 schema-backed report로 남긴다.
   - `outcome-metrics.json`은 session rollup과 run telemetry를 합쳐 rework count, HOLD/DISCARD moving average, rollback signal, operator effort proxy, defect escape proxy를 audit-oriented report로 남긴다. 이 값은 아직 `mechanism_review` priority, promotion gate, release gate 판정을 바꾸지 않는다. 단, `mutation_proposal`의 synthetic recent-log-overlap queue-unblock rotation은 같은 proposal의 최근 unresolved outcome이 남아 있으면 `recent_outcome_rework`로 runnable queue에서 내린다.
   - session report는 iteration 목록만이 아니라 routing/executor/telemetry rollup도 함께 담아, 어떤 role이 어떤 rung로 얼마나 dispatch됐고 어디서 block됐는지 세션 단위로 바로 읽을 수 있게 한다.

## 현재 상태

- `wiki/`는 일반 content corpus로 재분리됐다.
- `system/`은 self-improving maintainer 관련 지식을 담는다.
- `system/system-raw-registry.md`는 raw inventory summary router를 관리한다.
- `system/system-raw-registry/wiki.md`는 wiki raw inventory router이자 direct-entry shard고, 큰 family는 topic-specific second-order shard로 내려간다.
- `system/system-raw-registry/system.md`는 system corpus detailed registry shard다.
- raw registry는 `registry_id`를 machine key로 유지하고, `storage_path` 외에 optional `path_aliases`와 `content_sha256`를 통해 패키징/경로 drift를 흡수한다.
- `make refresh-generated` 또는 `.venv/bin/python -m ops.scripts.raw_registry_export --vault .`를 실행하면 `ops/raw-registry.json` export가 현재 `raw/` inventory를 기준으로 `path_aliases`와 `content_sha256`를 자동 파생해 채운다.
- `.venv/bin/python -m ops.scripts.raw_markdown_normalize --vault .`는 `raw/**/*.md`에 대해 minimal post-ingest normalization contract를 적용하고, rewrite 전후 sha256과 manual review flag를 report로 남긴다.
- `corpus_routing` 정책은 raw `type`과 `corpus`의 일치 여부를 lint fail로 강제한다.
- `system/system-log.md`는 전체 운영 이력을 append-only로 관리한다.
- promotion event는 `runs/<run-id>/promotion-report.json`으로 남기고, 필요할 때만 signoff와 `system/system-log.md` append를 연결한다.
- run closeout은 `runs/<run-id>/behavior-delta.json`과 `runs/<run-id>/run-artifact-fingerprint.json`도 함께 남겨 semantic/behavior review surface와 run-local artifact의 schema/size/hash를 고정하고, completed mechanism run은 `ops/reports/promotion-decision-trends.json`과 `ops/reports/outcome-metrics.json`을 갱신해 decision trend와 outcome proxy를 별도 report로 감시한다.
- `runs/<run-id>/runtime-events.jsonl`과 `ops/reports/runtime-events/<component>/*.jsonl`은 auto-improve/planning gate/executor/observability 경로의 structured execution log를 audit-only로 남긴다. 여기에는 `run_id`, `session_id`, `phase`, `component`, `decision`, `artifact_path`, `duration_ms`, `policy_version`이 들어가지만, canonical audit source는 계속 schema-backed JSON artifact다.
- `make outcome-metrics`는 cross-run `ops/reports/outcome-metrics.json`을 수동 갱신한다. 이 report는 `ops/schemas/outcome-metrics.schema.json`으로 검증되는 audit-only artifact다.
- `make supply-chain-provenance`는 `pyproject.toml`, `requirements.txt`, `requirements-dev.txt`, `uv.lock`, release manifest surface를 읽어 `ops/reports/supply-chain-provenance.json`을 만든다. 이 report는 `ops/schemas/supply-chain-provenance.schema.json`으로 검증되며, `ci_install_proof`, `lock_evidence`, `provenance_notes`를 함께 남긴다. `uv.lock`은 locked dependency evidence이고 dependency graph edge도 audit surface로 파싱하지만, CI install proof는 `.github/workflows/ci.yml`에서 구조화해 따로 기록한다.
- `make sbom-export-mapping`은 supply-chain provenance, release manifest, public export allowlist를 결합해 `ops/reports/sbom-export-mapping.json`을 만든다. 이 report는 dependency input이 release/public surface에 모두 닿는지와 public export가 release manifest의 subset인지 보여주는 audit artifact다.
- `make security-advisories`는 repo-native advisory input을 `ops/reports/security-advisories.json`으로 정규화하고, `make supply-chain-artifact-model`은 CycloneDX/OpenVEX/SPDX/in-toto/Sigstore가 공통으로 읽는 canonical graph를 `ops/reports/supply-chain-artifact-model.json`으로 고정한다.
- `make sbom-readiness-check`는 `ops/reports/sbom-readiness-gate-report.json`을 만들고, `make cyclonedx-sbom`은 `ops/reports/cyclonedx-bom.json`, `make spdx-sbom`은 `ops/reports/spdx-sbom.json`을 생성한다. 두 SBOM emitter 모두 같은 shared artifact model과 `uv.lock` dependency edge를 사용한다. CycloneDX는 표준 HTTP `$schema` URI를 유지하되 local schema alias로 vendored schema를 읽기 때문에 network access 없이 검증된다.
- `make openvex-draft`는 repo-native advisory input이 있으면 실제 statement를 authoring하고, 없으면 `vulnerability_source: not_scanned`인 draft shell을 남긴다. 이 OpenVEX draft는 repo-local schema를 유지하면서 top-level canonical artifact envelope/currentness metadata도 함께 기록한다. `make openvex-draft-cached`는 security advisories, model, CycloneDX, SPDX, OpenVEX, in-toto/SLSA statement, Sigstore verification metadata를 한 프로세스에서 재사용하며 생성하는 canonical fast path다.
- `make in-toto-statement`는 `ops/reports/in-toto-statement.json`에 SLSA provenance predicate를 포함한 in-toto Statement를 남기고, `make sigstore-bundle`은 `ops/reports/sigstore-bundle-verification.json`에 local integrity check와 optional external bundle presence를 고정한다.
- `make supply-chain-check`와 alias인 `make provenance-check`는 이 report를 strict gate 후보로 평가해 `ops/reports/supply-chain-gate-report.json`을 쓴다. required input missing, parser error, malformed `uv.lock`, CI install proof drift는 여기서 fail로 승격되지만, 기본 `release-check`와 `release-clean`은 그대로 유지한다.
- `make release-provenance-clean`과 `make release-sbom-clean`은 optional strict target이다. 즉시 default release gate로 승격하지 않고, 별도 clean surface로만 연결하는 것이 현재 canonical 운영 계약이다.
- auto-improve executor는 `auto_improve_policy.allowed_executors`에 있는 값만 허용하며, 현재 canonical runtime은 `codex_exec` 경로로 고정된다.
- mechanism review queue는 `ops/reports/mechanism-review-candidates.json` 같은 generated artifact에 두고, `system/system-log.md`는 계속 채택된 실험 chronology만 관리한다.
- mechanism review candidate는 historical calibration 외에도 auto-improve session rollup을 읽은 `session_calibration`을 optional하게 붙일 수 있고, 이때 우선순위 보정은 validation/review/mutation outcome, validator/reviewer dispatch, high-risk routing만 사용한다. 시간 합계나 model/reasoning mix는 prioritization 입력에서 제외한다.
- auto-improve session의 canonical envelope는 `ops/reports/auto-improve-sessions/<session-id>.json`이다. 각 `iterations[]`는 proposal/scope/routing/executor/run/promotion artifact ref와 normalized `decision`/`outcome`/`failure_taxonomy`/`quarantined`/`primary_targets`를 담고, top-level `learning_summary`가 attempt/rework/rollback/defect-escape/session-context evidence를 고정한다.
- mechanism review report top-level `diagnostics.session_calibration`은 optional candidate field와 별개로 항상 존재하는 feature-state envelope다. `enabled: true|false`와 `status: disabled|no_candidates|no_session_context|active`를 함께 담아, 기능이 꺼진 경우와 세션 문맥이 비어 있는 경우를 구분한 채 family aggregate를 보여준다.
- mechanism review report top-level `diagnostics.outcome_metrics_calibration`은 `ops/reports/outcome-metrics.json`의 rework/HOLD/DISCARD/rollback/defect-escape proxy를 family와 primary target별로 보여주는 audit-only preview다. `mode: audit_only`와 `gate_effect: none`으로 고정되며, 이 단계에서는 candidate priority, mutation proposal priority, promotion/release gate 판정을 바꾸지 않는다. 별도로 `mutation_proposal`은 반복 HOLD/DISCARD된 synthetic queue-unblock rotation을 runnable로 재발행하지 않기 위해 proposal-local recent outcome guard만 적용한다.
- canonical policy는 session calibration을 기본 `enabled: true`로 두지만, 필요하면 policy override에서 `mechanism_review.calibration.enabled: false`로 꺼서 historical calibration만 사용하게 만들 수 있다.
- mutation proposal layer는 `ops/reports/mutation-proposals.json`에서 mechanism review candidate를 단일 실험 가설로 좁히고, `priority_breakdown`으로 base/historical/session/log-overlap penalty를 함께 남겨 실제 queue order를 설명한다. `system/system-log.md`는 여전히 queue가 아니라 chronology로만 유지한다. Timestamped log overlap은 configured max-age window가 지나면 expired로 보아 `diagnostics.recent_log_overlap.scanned_log_headings`에서 빠지고, 같은 target이 count-window에만 남아 있다는 이유로 runnable proposal을 막지 않는다.
- mutation proposal report의 `diagnostics.family_session_calibration`은 mechanism review의 family-level calibration summary를 proposal queue 관점으로 얇게 복사해, family별 proposal 수·blocked proposal 수·session priority delta를 review report 없이도 바로 읽을 수 있게 한다. 이 block도 `enabled`와 `status`를 함께 가져가므로 upstream calibration이 꺼진 경우를 명시적으로 드러낸다.
- mutation proposal report의 `summary.queue_pressure_summary`는 그 family aggregate 중 상위 pressure만 한 줄로 압축해 보여준다. diagnostics를 열지 않아도 어떤 family가 proposal 수, blocked 상태, session delta 때문에 위아래로 움직였는지 빠르게 읽을 수 있고, feature가 꺼져 있으면 disabled 상태를 한 줄에서 바로 드러낸다.
- future buggy run 처리 기본값은 physical relocation이 아니라 promotion report `history.status` 갱신이다. directory move는 예외적인 long-term archival cleanup일 뿐 active-history exclusion의 canonical mechanism이 아니다.
- `system_mechanism` promotion은 기본적으로 eval improvement를 우선하지만, same-eval에서도 lint/complexity/tests secondary axis가 모두 비회귀이고 최소 1축이 엄격히 개선되면 signoff 하에 promotion할 수 있다.
- 자동 `system_mechanism` 루프에서는 promotion policy가 signoff-free override를 사용한다. `signoff_defaults.system_mechanism = not_required`, `hold_on_signoff_pending = false`, `promote_only_if_signoff_approved = false`를 기준으로 repo health와 promotion artifact가 곧 governing surface가 된다.
- 이때 complexity axis는 primary target만 얇아진 것을 곧바로 개선으로 보지 않고, supporting target까지 합친 `total_structural_metrics` 비회귀/개선을 기준으로 읽는다.

## Runtime 분리 현황 요약

- 상세 현황은 `ops/README.md`의 `Runtime 분리 현황`을 canonical 운영 메모로 본다.
- 이미 분리된 핵심 표면은 `run_mechanism_experiment_runtime.py`의 helper 모듈화, auto-improve session rollup/proposal queue/execution outcome 분리, `run_auto_improve_session()` phase helper, promotion gate state/rule/report helper, mechanism review history/candidate 분리, scaffold template payload 분리, scaffold command/input resolution 분리, registry lint pass support/alignment/review-candidate 분리, policy safety invariant/registry validation 분리다.
- 이미 닫힌 안전 경계는 manifest apply의 resolved containment와 symlink segment reject, explicit canary apply plan, shadow apply report, explicit live apply rollback rehearsal, atomic multi-write의 canonical duplicate target hard fail이다.
- 여전히 미완인 표면은 `mechanism_run_scaffold_runtime.py`의 starter/scaffold-load orchestration, downstream promotion decision registry alignment, content-addressed run-local provenance snapshot 보강이다. repo-native provenance gate, shared artifact model, CycloneDX/SPDX, OpenVEX authoring, in-toto/SLSA, Sigstore verification metadata, PyPI Trusted Publishing release workflow는 올라왔지만, default release hard gate 승격 여부는 아직 운영 판단 과제로 남아 있다.
- module surface contract는 `pyproject.toml`의 `[project.scripts]`, `ops/script-output-surfaces.json`의 `direct_fallback_eligible`, `ops/script-module-surfaces.json`으로 나뉜다. 설치형 canonical CLI는 `llm-wiki-*` console script이고, direct 실행 파일은 repo-local compatibility를 위한 transitional wrapper로만 유지한다. `ops/script-output-surfaces.json`는 AST 스캔으로 `if __name__ == "__main__"` 블록과 `--out`/`*-out` option, `resolve_repo_output_path` 사용 여부를 자동 감지하며, `direct_fallback_eligible` 필드로 direct script fallback 대상을 생성형 registry로 관리한다. `ops/script-module-surfaces.json`은 안정 import surface와 `__all__` export 목록을 고정하며, public tests가 세 계약의 drift를 함께 잡는다. output writer surface는 `ops/scripts/script_output_surfaces.py`가 생성하는 schema-backed canonical artifact envelope `ops/script-output-surfaces.json`를 사용해 `--out`/`*-out` option과 permissive/repo resolver 사용 drift를 함께 검증하며, candidate/promote 경로에서 artifact_kind, producer, source_tree_fingerprint, currentness를 고정한다.
- 다음 개선은 promotion decision registry의 downstream schema/reducer/finalize 정렬과 scaffold starter/load orchestration 분리 필요성 재평가를 1순위로 둔다. auto-improve phase helper의 파일 단위 이동은 현재 즉시 과제가 아니라, 독립 fault injection이나 외부 재사용 필요가 생길 때 진행한다.

## Project-scoped subagents

- repository-shared Codex subagent profile은 `.codex/agents/` 아래에 둔다.
- core role은 `explorer.toml`, `worker.toml`, `reviewer.toml`, `validator.toml`이다.
- 기본 ladder는 `gpt-5.5`+`medium` -> `gpt-5.5`+`high` -> `gpt-5.5`+`xhigh`의 3단계만 사용한다.
- `.toml`의 model/reasoning은 repo-shared fallback이고, 실제 배정은 `ops/scripts/select_subagent_rung.py`가 task complexity와 risk를 보고 같은 ladder 안에서 고른다.
- 기본 report 경로는 `ops/reports/subagent-routing-report.json`이고, `primary/supporting/test target + manual risk flag`를 입력으로 받는다.
- 자동 개선 루프에서는 selector를 role별로 반복 호출해 `runs/<run-id>/subagent-routing.worker.json`, `subagent-routing.reviewer.json`, `subagent-routing.validator.json`, 필요 시 auditor routing artifact를 남긴다.
- 실제 executor는 `codex exec` adapter를 사용하고, `.toml` intent를 role prompt로 materialize한 뒤 `<role>-executor-report.json`으로 결과를 고정한다. role별 `codex exec` timeout과 wrapper mutation/check command timeout은 `auto_improve_policy.defaults`에서 관리되고, timeout 여부는 executor report, run telemetry, `*-timeout-failure.json` artifact에 남긴다.
- public mirror에도 `.codex/agents/`를 포함하므로, subagent routing/exec 경로를 그대로 재현할 수 있다.
- 예시:
  - `.venv/bin/python -m ops.scripts.select_subagent_rung --vault . --role worker --primary-target ops/scripts/promotion_gate.py --supporting-target ops/policies/wiki-maintainer-policy.yaml --test-file tests/test_promotion_gate_equal_score.py`
- `worker.toml`이 기본 구현자이고, `validator.toml`은 bounded code path의 기본 validator다. 자동 실행에서는 pytest/tempfile 검증을 위해 temp workspace `workspace-write` sandbox를 쓸 수 있지만, executor가 non-worker source/control file mutation을 차단한다.
- mechanism experiment workspace는 live repo `.venv`를 temp workspace에 symlink해 network 없이도 같은 dependency set으로 worker/validator 검증을 수행한다. `.venv`는 여전히 diff/apply 대상에서 제외된다.
- provenance, replay parity, benchmark evidence, owner boundary, policy-weight 감사는 specialized add-on을 붙이고, 그래도 generic review/validation은 생략하지 않는다.
- 사용 기준과 각 프로필의 목적은 `.codex/agents/README.md`에 정리했다.

## 메타데이터와 Obsidian

- `wiki/`와 `system/`의 모든 page는 YAML frontmatter를 가진다.
- 공통 속성은 `title`, `page_type`, `corpus`, `aliases`, `tags`다.
- source page는 `registry_id`, `raw_path`, `source_type`, `domain`을 추가로 가진다.
- `source_type: domain-research-paper` page는 `research_mode: experiment|model|reference`도 함께 가진다.
- concept page는 `canonical: true`를 사용한다.
- synthesis / query page는 `source_count`를 둘 수 있다.
- reserved page는 `special_role`로 구분한다.
- Obsidian에서는 `All Properties`, `Bases`, `tag pane`를 켜 두고 `corpus/*`, `type/*` 태그와 properties를 기준으로 탐색하는 흐름을 권장한다.
