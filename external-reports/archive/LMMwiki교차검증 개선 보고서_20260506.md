## 1. 최종 요약

두 신규 리뷰는 서로 다른 문서 형식을 취하지만, 실제로는 같은 체크포인트인 ZIP SHA `390a3943890868b56e5ab1bdbdb9249d1b426b2acd8d91828a4891a0572f7271`을 대상으로 같은 결론에 도달한다. 핵심 판정은 다음이다.

 기존 리뷰가 지적했던 배포 오염, fresh extract `mtime` 오탐, current distribution ZIP 식별, same-eval telemetry discoverability, learning-delta-scoreboard 부재 문제는 코드·스키마·Makefile·테스트 표면에서 상당 부분 반영되었다. 그러나 현재 ZIP은 `opsreports`, `external-reports`, `runs`를 의도적으로 제외한 sourcecontent package에 가깝기 때문에, 이 ZIP 하나만으로 clean release, machine release, learning readiness, full-suite 대표성을 주장할 수 없다. 다음 핵심 작업은 새 기능 추가가 아니라, 현 ZIP SHA에 결박된 Evidence Companion Bundle을 생성하고 그 산출물에서 releasetestlearning 상태를 재판정하는 것이다.

이번 실제 파일 대조 결과, 두 신규 리뷰의 큰 결론은 대부분 타당하다. 다만 일부 표현은 실제 파일 기준으로 보정해야 한다.

1. 현 ZIP에는 `tmp`, `runs`, `opsreports`, `external-reports`, `.obsidian` 파일이 0개다. 이전 ZIP의 배포 오염 문제는 현재 source ZIP 표면에서는 제거되었다.
2. ZIP member timestamp는 모두 1980-01-01로 정규화되어 있다. 따라서 `zip_member_timestamp` basis는 fresh extract filesystem mtime 오탐을 피하는 보조 수단으로는 유효하지만, source freshness의 시간 증거로 쓰기에는 부적합하다. freshness authority는 digestfingerprint 중심으로 전환해야 한다.
3. `release_closeout_batch_manifest.py`에는 `--zip-metadata`, `--zip-timestamp-timezone`, `zip_member_timestamp` basis, `missing_zip_member_count` 등이 구현되어 있다.
4. `external_report_reference_manifest.py`에는 `--current-distribution-zip-path` 기반 current distribution ZIP identity 계산 경로가 있다. 하지만 현 ZIP에는 canonical `external-reportsreport-reference-manifest.json`이 없다.
5. `auto_improve_iteration_persistence_runtime.py`와 `run-telemetry.schema.json`에는 `observed_at`, `behavior_delta_digest`, `same_eval_reason`, `proposal_id`, `source_candidate_id`가 반영되어 있다. 그러나 `auto_improve_readiness_runtime.py`가 이 새 필드를 실제 판정에 소비한다는 증거는 아직 약하다.
6. `learning_delta_scoreboard.py`, schema, 테스트, Makefile target은 존재한다. 하지만 `release-evidence-closeout` recipe에는 `learning-delta-scoreboard` 호출이 포함되어 있지 않아 release gate blocking evidence로 자동 소비되지 않는다.
7. `test-execution-summary-full` target은 존재하고 `release-evidence-closeout`에 연결되어 있다. 그러나 현 ZIP에는 `opsreportstest-execution-summary-full.json`과 shard artifact가 없다.
8. 두 신규 리뷰 중 하나가 언급한 `_artifact_digest_mismatches()`의 중복 `for item in current.get(artifacts, [])` 문제는 현재 실제 파일에서는 발견되지 않았다. 해당 버그는 이미 수정되었거나 리뷰 작성 시점의 다른 파일에서 나온 지적으로 보인다.
9. 이번 환경에서 `build_batch_manifest(... zip_metadata_path=...)` 전체 직접 실행은 300초 내 종료되지 않았다. 따라서 이 보고서는 builder pass를 신규 실행 증거로 주장하지 않고, 실제 코드·스키마·Makefile 표면과 두 리뷰의 실행 기록을 분리해 기록한다.

## 3. 두 신규 리뷰의 관계와 차이

### 3.1 신규 리뷰 1 `llmwiki_unified_crosscheck_report.md`

신규 리뷰 1은 세 병렬 리뷰의 공통 결론을 통합한 형태다. 장점은 다음이다.

- 현재 ZIP을 source package로 판정하고, 별도 Evidence Companion Bundle 필요성을 가장 선명하게 고정한다.
- 판정 축을 packaging hygiene, release manifest, deterministic packaging, ZIP metadata freshness basis, current distribution identity, auto-improve telemetry, learning-delta-scoreboard, full-suite evidence, release authority로 분리한다.
- P0 잔여 작업을 canonical evidence companion bundle, digestfingerprint freshness, current distribution manifest, run telemetry consumer 연결, full-suite artifact, current distribution Makefile hard requirement, archive profile manifest로 체계화한다.
- source ZIP과 evidence bundle의 이중 산출물 계약을 명확히 제안한다.

보정할 점은 다음이다.

- `release_closeout_batch_manifest.py`의 중복 loop 문제는 현재 실제 파일에서는 발견되지 않았다. 리뷰 작성 당시의 중간 파일이나 이전 상태에 대한 지적이었을 가능성이 있다.
- batch manifest builder pass는 리뷰 자체의 실행 기록으로는 유효하지만, 이번 재실행에서는 장시간 종료되지 않아 새 증거로 재확인하지 못했다.

### 3.2 신규 리뷰 2 `LLMwiki 통합 리뷰 보고서.md`

신규 리뷰 2는 실제 파일 대조와 코드 수준 세부 지적이 더 강하다. 장점은 다음이다.

- `release_closeout_batch_manifest.py`, `external_report_reference_manifest.py`, `auto_improve_iteration_persistence_runtime.py`, `learning_delta_scoreboard.py`, `Makefile`의 실제 반영 항목을 구체적으로 열거한다.
- ZIP timestamp가 모두 1980으로 정규화된 상황에서 `zip_member_timestamp`를 freshness temporal proof로 쓰기 어렵다는 점을 명확히 지적한다.
- `artifact_freshness_runtime.py`의 root-prefixed ZIP path matching, `learning_delta_scoreboard._coverage()` denominator 0 처리, hardcoded routing aggregate path 등 구체적 개선 지점을 제시한다.
- full-suite evidence와 targeted summary 분리를 구현 표면과 artifact 부재로 나누어 평가한다.

보정할 점은 다음이다.

- `_artifact_digest_mismatches()`의 중복 loop는 실제 현재 파일에서 수정된 상태다. 따라서 최종 보고서에서는 “확인 필요”가 아니라 “현재 파일 기준 발견되지 않음”으로 판정해야 한다.
- builder 직접 실행 pass, 일부 테스트 hang환경 문제는 리뷰의 실행 환경 기록으로 남기되, 이번 재검토에서 전체 재현된 사실과 분리해야 한다.

### 3.3 기존 기준 리뷰와의 연결

기존 기준 리뷰는 직전 ZIP(SHA `aef28bb8...`)에서 다음 문제를 확인했다.

- ZIP entryfiledir 수 1,839  1,740  99.
- `tmp` 파일 7개 포함.
- `external-reportsreport-reference-manifest.json`이 과거 ZIP을 current distribution으로 참조.
- fresh extract에서 batch manifest check가 filesystem `mtime`에 의해 실패.
- `auto-improve-readiness.json`의 telemetry coverage 0 및 `can_promote_result=false`.
- `test-execution-summary.json`은 report-contract subset이며 full-suite evidence가 아님.
- cleanmachine release가 아니라 conditionaloperator release 상태.

현 ZIP은 이 중 배포 오염 제거, source manifest 일치, ZIP metadata basis 구현, telemetry writerschema 보강, scoreboard 도입은 상당 부분 반영했다. 반면 canonical evidence artifact 생성, current ZIP SHA closure, full-suite summary 봉인, readiness consumer 연결, release authority 재판정은 아직 남아 있다.

## 7. 주요 컴포넌트별 상세 대조

### 7.1 Release package hygiene

검증 결과 완료에 가까운 개선이다.

현 ZIP은 source manifest와 완전히 일치하며, 직전 ZIP에서 문제가 되었던 `tmp`, `runs`, `external-reports`, `opsreports`, `.obsidian` 혼입이 없다. 이 점은 두 신규 리뷰와 실제 ZIP이 일치한다.

주의점 `opsreports`와 `external-reports`가 제외된 것은 stale evidence 혼입을 막는 데 유리하지만, 동시에 current release evidence를 ZIP 내부에서 확인할 수 없게 만든다. 따라서 source ZIP과 evidence bundle의 관계를 명시하지 않으면 “깨끗한 source package”가 “검증 완료 release”로 오해될 수 있다.

### 10.2 Timestamp semantics 분리

현재 `zip_member_timestamp`는 1980 정규화값이다. 따라서 다음처럼 semantic field가 필요하다.

```json
{
  source_evidence_freshness {
    basis zip_member_timestamp,
    timestamp_semantics normalized_archive_timestamp,
    normalized_timestamp_policy all_zip_members_set_to_1980_01_01_00_00_00,
    source_freshness_temporal_authority not_claimed,
    source_freshness_content_authority source_tree_fingerprint,
    changed_after_generated_at_count 0,
    missing_zip_member_count 0
  }
}
```

### 10.3 Digestfingerprint 중심 freshness 모델

추천 판정 기준

1. ZIP member path set == source manifest path set.
2. ZIP member content digest == source manifest digest.
3. Evidence artifact `input_fingerprints`가 source manifest digest와 일치.
4. Generated report `generated_at`은 시간 증거가 아니라 산출물 ordering metadata로만 사용.
5. Normalized timestamp archive에서는 `changed_after_generated_at_count`를 freshness authority로 쓰지 않음.

### 10.4 Typed same-eval reason contract

현재 free-text `same_eval_reason`은 다음 구조로 확장하는 것이 바람직하다.

```json
{
  same_eval {
    baseline_eval 4792,
    candidate_eval 4792,
    reason_code telemetry_discoverability_improved,
    strict_secondary_improvement_present true,
    secondary_improvement_axes [run_telemetry_discoverability, behavior_delta_digest_coverage],
    promotion_allowed true
  }
}
```

권장 enum

 code  의미  promotion 
 ---  ---  --- 
 `candidate_eval_improved`  candidate eval  baseline eval  allowed 
 `telemetry_discoverability_improved`  동점이나 telemetry recoverability 개선  conditional allowed 
 `behavior_delta_digest_added`  동점이나 digest evidence 추가  conditional allowed 
 `same_eval_no_secondary_improvement`  동점, secondary 개선 없음  blocked 
 `noop_mutation`  실질 변화 없음  blocked 
 `insufficient_benchmark_resolution`  벤치마크 해상도 부족  review required

### 10.5 Proposal-family aware aggregate selector

 coverage 종류  의미

## 13. 최종 판정

두 신규 리뷰는 기존 리뷰의 문제의식을 대부분 정확히 계승하고 있으며, 현재 실제 ZIP과 대조해도 큰 방향은 타당하다. 차이는 다음 한 문장으로 요약된다.

 직전 리뷰 시점의 문제는 “배포물 안에 stalegenerated evidence와 tmp가 섞여 있고, mtime 기반 검증이 fresh extract에서 깨지는 것”이었다. 현 시점의 문제는 “source package는 깨끗해졌지만, current ZIP SHA에 묶인 canonical evidence bundle이 없어 releaselearningfull-suite closure를 아직 주장할 수 없는 것”이다.

따라서 지금 가장 중요한 작업은 새 리포트를 더 쓰는 것이 아니다. 이미 추가된 코드 표면을 실제 산출물 생성·봉인·소비 경로까지 연결해야 한다.

우선순위는 다음과 같다.

1. Evidence Companion Bundle 생성 현 ZIP SHA `390a3943...` 기준 releasetestlearning evidence를 별도 bundle로 봉인한다.
2. DigestFingerprint Freshness 전환 1980 timestamp 정규화 조건에서 temporal freshness 주장을 중단하고 content authority로 이동한다.
3. Telemetry Consumer 연결 writer가 기록하는 `same_eval_reason`, `behavior_delta_digest`, `observed_at`을 readinessoutcomemutation proposal queue가 실제로 소비하게 한다.
4. Scoreboard Gate 통합 learning-delta-scoreboard를 release closeout에 포함하고, no-evidence와 full-coverage를 구분한다.
5. Full-suite Artifact Closure full-suite summary와 shards를 생성해 targeted report-contract summary와 완전히 분리한다.

이 다섯 가지가 닫히기 전까지는 “개선 코드가 들어갔다”는 말은 가능하지만, “clean release ready”, “machine release allowed”, “learning readiness recovered”, “full-suite representative evidence available”이라는 주장은 보류해야 한다.
