# ops/

이 디렉터리는 wiki maintainer runtime을 **문서 밖으로 외부화**한 control layer다.

## 목적
- wiki와 system corpus를 평가하는 binary eval suite 제공
- lint / eval / manifest 생성 자동화
- PASS / WARN / FAIL 기준을 policy 와 schema 로 고정
- future meta-loop가 수정해야 할 target을 명확히 분리

## 구성
- `policies/wiki-maintainer-policy.yaml`
- `evals/wiki-quality-evals.md`
- `schemas/*.json`
- `templates/*.yaml|json`
- `scripts/*.py`
- `../requirements.txt`
- `../requirements-dev.txt`
- `../pytest.ini`
- `../pyproject.toml`

- `make dev-install`은 `uv`가 있으면 `.venv`를 `uv venv`로 만들고 `requirements-dev.txt`와 editable install `-e .`를 함께 고정한다.
- `uv`가 없으면 같은 target이 `python3 -m venv .venv`, `.venv/bin/python -m pip install -r requirements-dev.txt`, `.venv/bin/python -m pip install -e .` 순서로 fallback한다.
- `make bootstrap-preflight`는 repo가 선택한 `$(PYTHON)`에서 Python 3.12+, PyYAML/jsonschema, pytest/ruff/mypy가 보이는지 확인하고 `ops/reports/bootstrap-preflight-report.json`에 schema-backed bootstrap evidence를 남긴다. dependency가 빠져 있으면 `make dev-install`로 복구하라는 machine-readable/text 진단을 낸다. report의 `environment` block은 `environment_class`, dependency source, install attempt 여부, interpreter selection을 기록해 developer, clean release-builder, review container evidence를 분리한다. 모듈 직접 실행 경로인 `python -m ops.scripts.bootstrap_preflight --dev --json`은 stdlib-only 진단을 유지해 system python dependency gap도 raw import error 없이 보여 준다.
- 일회성 실행만 필요하면 `uv run --with pytest --with pytest-xdist --with ruff --with mypy --with types-PyYAML --with jsonschema --with pyyaml make check`처럼 ephemeral path를 써도 된다.
- `make`를 바로 쓰기 어려운 환경에서는 `.venv`를 만든 뒤 `uv sync --extra dev` 또는 동일한 `pip install -r requirements-dev.txt` + `pip install -e .` 조합을 공식 대체 경로로 본다.
- `planning_gate_validate.py`는 `--artifact-dir`를 생략하면 policy에 선언된 기본 starter bundle path를 사용한다.
- `make sanitize-runs`는 기존 `runs/` artifact 안의 절대경로와 temp workspace 경로를 repo-relative 형태로 다시 정리하는 공개 전 hygiene helper다.
- `make public-export`는 code/ops-only 공개 미러를 기본적으로 `/tmp/llm-wiki-public-repo/`에 만들고, `make test-public`는 현재 작업 트리에서 `public` marker 테스트만 빠르게 돌린다. `make public-check`는 그 exported tree 안에서 static gate와 같은 `public` tier를 다시 실행하고 `ops/reports/public-check-summary.json`에 schema-backed evidence를 promote한다. exported tree 전체 스위트가 필요하면 `make public-check-all`을 사용한다.
- `make sync-public-policy`는 `ops/scripts/public_surface_policy.py`를 source of truth로 삼아 루트 `.gitignore`의 public allowlist block을 다시 맞춘다.
- 루트 `.gitignore`는 같은 public surface policy를 따라 private corpus, raw inventory, generated report가 기본 추적 대상에 들어오지 않게 맞춘다.
- 공개 미러는 `raw/`, `wiki/`, `system/`, `runs/`, `external-reports/`, `ops/reports/`, generated registry/manifest artifact를 포함하지 않는다.
- `.codex/agents/`는 공개 미러에 포함해 subagent routing과 executor path가 code/ops mirror에서도 그대로 재현되게 한다.
- root `AGENTS.md`는 public-safe base instruction이고, full local canonical vault supplement는 `AGENTS.local.md`에 둔다.
- private canonical vault의 `ops/raw-registry.json`과 `ops/manifest.json`은 private corpus inventory를 담기 때문에 제외하고, 대신 exported tree에는 `PUBLIC-EXPORT-MANIFEST.json`만 남긴다.
- 구조 설명의 public-safe canonical surface는 루트 [ARCHITECTURE.md](../ARCHITECTURE.md)다. private `system/system-index.md`는 router 역할에 집중한다.

## 현재 계약
- `ops/policies/wiki-maintainer-policy.yaml`은 runtime policy의 single source of truth이고, loader는 mirrored Python default policy 없이 이 파일과 schema를 직접 검증한다.
- YAML parsing은 `PyYAML` 기반 `yaml_runtime.py`를 사용하며, frontmatter / policy / planning artifact가 같은 loader를 공유한다.
- `wiki_lint.py`와 `wiki_eval.py`는 `wiki/`와 `system/`을 함께 스캔한다.
- `wiki_eval.py`는 Stage 1 contract eval이고, `wiki_stage2_eval.py`는 source-count consistency / central research anchor layer / broad synthesis boundary / source-only seed absorption hint를 보는 Stage 2 semantic-operational eval이다.
- `wiki_snapshot_runtime.py`는 page discovery, text, frontmatter, wikilink resolution, optional source-trace resolution state를 reusable in-memory snapshot으로 묶는다. `wiki_lint.py`, `wiki_eval.py`, `wiki_stage2_eval.py`는 optional shared snapshot을 받을 수 있고, same-process caller는 이 계층으로 중복 I/O를 줄일 수 있다.
- Stage 2 eval은 이제 `make check`의 기본 gate에 포함된다.
- tiered test file은 각 module의 `pytestmark`로 registry-defined lane marker를 직접 선언한다. `tests/conftest.py`는 자동 분류 대신 `ops/test-lane-registry.json`을 읽어 허용/금지 marker 조합만 검증한다.
- `make static`은 `ops/scripts`, `tests`, `tools` 전체를 ruff hard gate로 검사하고, `MYPY_TARGETS` allowlist만 mypy hard gate로 검사한다. 기본 mypy allowlist는 `ops/mypy-allowlist.txt`에 둔 전체 `ops/scripts/*.py` surface다.
- `make fast-smoke`는 subagent/developer precheck 전용 curated smoke다. strict import fallback/module-surface contract, frozen report schema samples, artifact I/O helper, mechanism review runtime, mutation proposal primary-evidence fail-fast edges, auto-improve readiness runtime, artifact freshness, release-smoke unit tests를 같은 pytest invocation으로 묶되, full wiki lint/eval, `integration-heavy` repo-root generator smoke, full `make release-smoke` package gate는 제외한다.
- `make test-fast`가 기본 fast pytest lane이며, 기존 `make test`와 `make unit-tests`는 compatibility alias다. Report lane은 `make test-report-contract`, checked-in generated artifact final self-check는 `make test-artifact-finalization`, release sealing은 `make test-release-sealing`, real subprocess/startup/timeout surface는 `make test-subprocess`가 담당한다.
- `make report-contracts`는 report/schema/generated-artifact 변경 직후 쓰는 targeted core verification path다. 기본값은 wrapper/self-reference 계약, Makefile/static workflow, checked-in report contract, schema sample, template contract처럼 report evidence를 닫는 데 직접 필요한 테스트만 직렬 실행한다. 운영 report generator와 release aggregator의 긴 회귀 검증은 `make report-contracts-extended`에서 명시적으로 실행한다.
- `make test-execution-summary`는 report-contract pytest invocation을 schema-backed `ops/reports/test-execution-summary.json`으로 고정한다. 명시 target인 `make test-execution-summary-report-contract`도 같은 lane을 실행하고, `make test-execution-summary-fast`와 `make test-execution-summary-full`은 각각 fast/full pytest evidence를 별도 report로 남긴다. 현재 full-suite refresh lane은 `make test-execution-summary-full-refresh`이며 refreshed evidence에 맞춘 `TEST_EXECUTION_SUMMARY_FULL_EXPECTED_NODE_COUNT=1257`를 operator-visible 기준으로 노출하고, refresh 직후 collect-only `nodeid_count`가 이 기준과 다르면 실패한다. full-suite canonical report는 `ops/reports/test-execution-summary-full-shards/*.json` shard를 다시 읽어 집계한 aggregate authority이고, failed node ids는 `build/release-payloads/test-execution-summary-full.failed-nodeids.txt` evidence artifact로 분리한다. 이 summary는 pytest selector에서 해석한 대상 test 파일의 `test_target_fingerprints`와 optional collect-only node id digest를 함께 기록하며, artifact freshness는 대상 test 파일 mtime이 summary `generated_at`보다 최신이면 summary를 stale로 분류한다. `execution_environment`는 Python version, pytest version, `PYTEST_DISABLE_PLUGIN_AUTOLOAD` policy, wrapped command interpreter path class를 구조화해 `.venv` 없는 exported ZIP에서도 실행 조건 차이를 비교하게 한다. summary lane은 `REPORT_CONTRACT_SUMMARY_MARK_EXPR=report_contract and not artifact_finalization`으로 checked-in generated artifact self-check를 제외하고, 해당 일곱 검사는 generated artifact refresh 뒤 `make test-artifact-finalization` 및 selector-free full-suite refresh에서 실행한다. 기존 `make report-contract-finalization`은 compatibility alias다. pytest `--deselect` 항목은 `deselected_tests[]`로 구조화하지만 `ops/policies/report-contract-deselections.json`은 max_count 0의 empty policy라서 새 deselection이 생기면 `deselection_lifecycle`과 clean lane contract가 즉시 실패한다. `make test-execution-summary-reuse`는 기존 passing summary가 현재 command, target file fingerprint, zero structured deselection, lifecycle, collect-only node id digest와 일치할 때 pytest wrapper를 다시 실행하지 않고 `summary_mode=reused` report를 갱신한다. 긴 report-contract wrapper 실행은 shard를 `ops/reports/test-execution-summary-shards/*.json`에 보존한 뒤 `make test-execution-summary-aggregate`로 집계할 수 있다. Aggregate 경로는 명시적 `--aggregate`에서만 동작하므로 shard 디렉터리가 있어도 일반 target은 pytest를 다시 실행한다.
- report-contract pytest invocation은 core lane과 extended lane으로 나뉜다. `REPORT_CONTRACT_TESTS`는 기본적으로 `REPORT_CONTRACT_CORE_TESTS`만 실행하며, 여기에는 wrapper/self-reference 계약, Makefile/static workflow, checked-in report contract, schema sample, template contract처럼 summary evidence에 직접 필요한 테스트만 들어간다. 40초 이상 걸리는 artifact freshness, archived-run backfill, raw-registry evidence bundle, release closeout/cohort, live report generation smoke는 `REPORT_CONTRACT_EXTENDED_TESTS`와 `make report-contracts-extended`에서 명시적으로 실행한다.
- source package clean lane은 `make test-source-package`와 `make release-source-package-check`가 담당한다. mark expression, deselection policy, deselect list, suite id는 registry의 `source_package` pack contract와 정적으로 맞춘다.
- `make report-contract-closeout`은 report/schema/generated-artifact 작업의 최종 증적 갱신 경로다. wrapper 전에 `release-smoke-full-reuse`로 현재 source tree 기준 full smoke 증거가 닫혔는지 확인하고, `report-contract-closeout-precheck`가 `ops/policies/report-contract-closeout.json`의 `pre_refresh_targets`를 읽어 `script-output-surfaces`, generated artifact index, auto-improve readiness, artifact freshness를 policy 순서대로 cheap precheck로 맞춘다. 이후 `test-execution-summary`로 비싼 report-contract pytest wrapper를 한 번만 실행한 뒤 generated artifact index, artifact freshness, write-only closeout summary, release evidence cohort를 정리한다. tmp advisory helper인 archive execution manifest는 canonical closeout bundle에 포함하지 않는다. closeout summary와 cohort를 쓴 뒤 freshness와 closeout summary를 cheap second pass로 한 번 더 맞추고 `test-artifact-finalization`을 실행해, generated artifact self-check와 `ops/script-output-surfaces.json` currentness drift가 refresh 이후 상태만 검증되게 한다.
- `make workflow-dependency-planner`는 변경된 path surface와 Make recipe 의존 관계를 schema-backed `tmp/workflow-dependency-planner.json`으로 정리한다. `WORKFLOW_DEPENDENCY_PLANNER_CHANGED_FILES_MANIFEST=<path>`를 주면 `changed_files[]` / `files[]` / `paths[]` manifest를 읽어 `report-contract-closeout`, `release-evidence-closeout`, `external-report-reference-manifest`, generated artifact finalization 같은 후속 target chain을 `selected_workflows[].steps[]`로 제안한다. `workflow_dependency_planner_closeout`의 fixed-point writer step은 `ops/policies/release-closeout-fixed-point.json`에서 동적으로 확장하며, `make release-workflow-order-guard`는 이 planner hook과 fixed-point writer order를 Makefile의 `release-evidence-closeout`/`check-finalized` recipe와 비교하는 non-mutating guard report를 `tmp/release-workflow-order-guard.json`에 쓴다. release closeout 자체는 이 advisory file들을 직접 canonical input으로 쓰지 않고, source-derived workflow graph와 guard runtime으로 ordering contract를 검증한다. `make workflow-dependency-planner-check`는 `tmp/workflow-dependency-planner-check.json`에만 진단을 쓴다.
- `make release-risk-taxonomy-matrix`는 `ops/policies/release-risk-taxonomy.json`에서 감사용 matrix를 재생성한다. canonical JSON은 `ops/reports/release-risk-taxonomy-matrix.json`, 사람이 바로 읽는 Markdown은 `ops/reports/release-risk-taxonomy-matrix.md`에 쓰며, 각 risk code의 clean/conditional/learning/advisory lane effect를 같은 source of truth에서 노출한다.
- `make release-evidence-closeout`은 final authoritative clean lane이다. Make prerequisite가 아니라 recipe 안에서 generated core/static/full smoke/source-package clean extract/report-contract summary를 닫고, 첫 `artifact-freshness` 전에 `public-check-summary`, `learning-claim-evidence-bundle`, `learning-confirmed-evidence-cohort`, `learning-claim-unlock-review`, `learning-delta-scoreboard`, write-only `release-closeout-summary-report`, `release-evidence-cohort`, write-only `release-evidence-dashboard-report`, `release-lane-summary`, `release-clean-blocker-ledger`를 한 번 pre-refresh해 schema-sensitive closeout JSON shape를 현재 generator와 맞춘다. 그 뒤 full-suite summary, report-contract summary, function-budget proposal, outcome/provenance gate policy, index/freshness를 다시 닫고 `public-check-summary`, learning evidence, strict `release-closeout-summary`, signoff revalidation, strict cohort, dashboard, lane summary, blocker ledger, artifact finalization을 실행한 뒤 `release-closeout-fixed-point`, final `external-report-action-matrix`, `tmp-json-clean`, `release-closeout-finality-verify`로 끝낸다. `make-target-inventory`, `workflow-dependency-planner`, `release-workflow-order-guard`, `archive-execution-manifest`는 tmp advisory outputs로 유지되고 canonical closeout writer set에는 직접 포함되지 않는다. `release-closeout-fixed-point`는 policy DAG의 writer set을 bounded iteration으로 수렴시키며, 첫 iteration에서만 generated-index heavy prerequisites를 실행하고 이후에는 changed downstream writer만 재실행한다. `release-closeout-batch-manifest.json`은 release evidence inventory와 release authority snapshot만 담당하고 finality digest를 직접 소유하지 않는다. fixed-point report는 writer set 수렴과 iteration별 digest map만 증명한다. 최종 digest binding 권위는 `ops/reports/release-closeout-finality-attestation.json`이며 fixed-point report digest, batch manifest digest, self-check digest, external report manifest digest, current tracked digest map을 단일 seal로 묶는다. distribution ZIP이 없거나 path/content parity가 맞지 않으면 semantic release가 clean이어도 batch manifest의 `sealed_release_status`는 `unsealed_distribution_not_provided`, `unsealed_distribution_invalid`, 또는 `unsealed_distribution_drift`로 남고 `sealed_clean_pass`가 되지 않는다. `make release-evidence-closeout-sealed`는 `release-distribution-zip`으로 `build/release/LLMwiki-source.zip`을 materialize하고 strict external report reference manifest를 같은 ZIP path로 갱신한 뒤, 같은 path를 `RELEASE_CLOSEOUT_DISTRIBUTION_ZIP` 및 `RELEASE_CLOSEOUT_BATCH_MANIFEST_ZIP_METADATA`로 넘겨 clean release authority와 materialized source package parity를 batch 안에 반영한다. full sealed target은 final rehearsal check output을 `build/release/release-closeout-sealed-rehearsal-check.json` sidecar로 써서 post-seal operator tmp JSON hygiene를 다시 dirty로 만들지 않는다. non-sealed closeout에서 current distribution ZIP이 없으면 external report strict evidence는 `external_report_strict_unavailable` typed attention으로 남고 sealed lane이 최종 해석한다. source ZIP self-description은 learning claim evidence bundle, confirmed evidence cohort, learning delta scoreboard를 pre-seal linked artifacts로 직접 기록한다. sealed lane은 이어서 `operator-release-summary`와 `release-post-seal-attestation`을 실행해 ZIP 밖 `build/release/release-post-seal-attestation.json` sidecar에 source ZIP, final batch manifest, self-check, operator summary, learning claim evidence bundle, confirmed evidence cohort, learning delta scoreboard digest를 non-cyclic하게 묶고, post-seal attestation은 source ZIP self-description에 이 learning links가 없으면 실패한다. full-suite JUnit/log raw payload는 기본 `RELEASE_AUDIT_PAYLOAD_STAGING_DIR=build/release-payloads`에 남겨 최종 `tmp-json-clean` 뒤에도 optional payload availability가 유지되며, `make release-audit-pack RELEASE_AUDIT_PACK_INCLUDE_OPTIONAL_PAYLOADS=1`을 지정할 때만 portable pack에 포함된다. `learning-delta-scoreboard`는 batch manifest나 operator summary를 input으로 읽지 않고, generated index/freshness도 final batch manifest와 closeout self-check를 sealing inventory에서 제외해 release authority digest cycle을 만들지 않는다. `make release-audit-pack`은 필요할 때만 batch manifest를 source of truth로 `tmp/release-audit-pack.zip`을 materialize하는 선택적 감사 view이고 새 release authority를 만들지 않는다. `make release-builder-full`은 이 closeout lane 전에 static과 full-suite test execution summary를 먼저 실행한다. 이 lane은 `machine_release_allowed=true`, strict same-fingerprint cohort, zero deselection, zero - 같은 report의 `summary.queue_pressure_summary`는 그 aggregate의 상위 pressure를 한 줄로 압축한 값이다. diagnostics를 펼치지 않아도 queue 상단에서 어떤 family가 proposal volume, blocked 상태, session calibration delta 때문에 움직였는지 바로 읽을 수 있고, calibration이 꺼진 경우도 summary에서 즉시 보인다.
- `run_mechanism_experiment.py --proposal-id ... --scaffold-only`는 generated proposal 하나를 선택해 `runs/<run-id>/proposal-snapshot.json`으로 freeze하고, seed/plan/open-questions/run-ledger/improvement-observations를 proposal-aware starter로 채운 뒤 멈춘다.
- `run_mechanism_experiment.py`는 optional evidence input으로 `--scope-freeze`, `--routing-report`, `--executor-report`를 받아 자동 루프가 남긴 bounded evidence를 그대로 흡수할 수 있다.

- `run_mechanism_experiment.py`는 mechanism-run starter를 scaffold하고, baseline/candidate lint·eval·mechanism assessment, repo health command, promotion gate, 조건부 finalization까지 한 번에 묶는 wrapper다.

- auto-improve session closeout은 `ops/reports/outcome-metrics.json`도 갱신한다. 이 report는 rework count, HOLD/DISCARD moving average, rollback signal moving average, operator effort proxy, defect escape proxy를 담는 audit-only cross-run outcome report이며, 아직 `mechanism_review` priority, promotion gate, release gate 판정에는 연결하지 않는다.
- `ops/reports/defect-escape-closures.json`와 `ops/reports/rework-closures.json`는 사람이 판단한 closure registry payload를 유지하되 `make closure-registry-envelope`가 canonical envelope, currentness, summary count를 재봉인한다. `generated-artifact-index`는 이 target을 prerequisite로 두어 envelope 누락 상태가 index/freshness 체인에 남지 않게 한다.
- executor는 `auto_improve_policy.allowed_executors` allowlist로 fail-closed 검증하며, 현재 자동 루프의 canonical executor는 `codex_exec`다.
- `auto_improve_policy.defaults.executor_timeout_seconds`는 role별 `codex exec` subprocess timeout이고, `wrapper_command_timeout_seconds`는 wrapper mutation/check command timeout이다. timeout은 blocking executor report, mutation/repo-health ledger decision, `run-telemetry.json`의 `command_timeouts`, 그리고 schema-backed `*-timeout-failure.json` artifact로 기록된다.
- `auto_improve_policy.apply_mode = live`가 canonical 기본값이다. PROMOTE는 기본 경로에서 bounded live repo mutation과 finalize까지 진행하며, live mutation 직전에는 `shadow-apply-report.json`과 `rollback-rehearsal-report.json`을 모두 남기고 rehearsal이 `pass`여야 한다. canary는 wrapper에서 `--apply-mode canary_only`를 명시한 경우에만 수행하며, 이 경로는 `workspace_apply_canary_ready` ledger event와 `shadow-apply-report.json`만 남기고 live repo mutate/finalize는 건너뛴다.
- 첫 live auto-improve run 전에는 `make auto-improve-readiness`, `make check-serial`, `make public-check-serial`을 순서대로 먼저 통과시키는 것을 canonical preflight로 본다. `make auto-improve-readiness`는 내부에서 `make refresh-generated-core`를 선행한 뒤 최신 queue inputs를 읽어 `ops/reports/auto-improve-readiness.json`을 만들고, emitted proposal 수가 아니라 실제 runnable proposal queue, `can_execute_trial`, `can_promote_result`, learning/promotion blocker를 기준으로 PASS/WARN을 판정한다. 테스트/운영 기준 entrypoint도 bare `pytest`가 아니라 `make` 또는 `.venv/bin/python -m pytest`다.
- readiness report의 canonical contract는 `execution_readiness`, `learning_readiness`, `can_execute_trial`, `can_promote_result`, `learning_blockers[]`, `promotion_blockers[]`, `release_blockers[]`다. CLI exit code와 preflight gate는 top-level `status`가 아니라 `execution_readiness.can_run`을 기준으로 판정한다. `learning_readiness.status`와 learning blocker id vocabulary는 `ops/scripts/learning_readiness_vocabulary.py` 및 `ops/schemas/learning-readiness-vocabulary.schema.json`의 shared contract를 따른다. 단, `learning_readiness.likely_to_learn`이 false이면 `learning_blockers[]`와 호환용 `release_blockers[]`에 `learning_blocked_by_*` named blocker가 남고, 실행 불가나 승격 불가 사유는 `promotion_blockers[]`에 모인다. `learning_blocked_by_review_required`는 learning improvement를 claim하는 release에서만 release closeout의 `learning-readiness-signoff` 경로로 수동 accepted risk 전환이 가능하며, signoff가 없거나 만료됐거나 template-only marker를 갖고 있으면 blocker로 남는다. `learning_blocked_by_execution_not_runnable`은 signoff로 수락되지 않고 live auto-improve readiness와 learning claim을 차단하지만 source package release 자체는 차단하지 않는다. top-level `next_action`도 execution-only 요약이 아니라 execution/learning 둘을 함께 반영하며, trial은 가능하지만 promotion이 막힌 경우에는 `Trial only; do not promote.` 문구로 승격 금지를 먼저 노출한다. queue layer의 `queue.ready`는 runnable proposal queue snapshot용 boolean으로만 유지하며, legacy `combined_state`와 `diagnostics.learnability_shadow_gate`는 제거됐다.
- `mutation-proposals.json` report selection은 unblocked proposal을 먼저 선택해 blocked proposal이 runnable 대안을 밀어내지 않게 한다. blocked proposal은 실행 큐에서는 제외되지만 `diagnostics.queue_selection.blocked_reason_counts`에 reason별 count로 남는다.
- `make auto-improve-readiness`가 WARN이면 auto-improve는 시작하지 않는다. `ops/reports/mutation-proposals.json`에 proposal이 있더라도 모두 `blocked_by` 상태면 still blocked로 본다. fallback family seed run이 아직 없을 때는 `ops/scripts/mechanism/auto_improve_iteration_persistence_runtime.py`와 `ops/schemas/run-telemetry.schema.json`을 대상으로 `tests/test_auto_improve_iteration_runtime.py`를 붙인 수동 `system_mechanism` run을 먼저 finalized하고, 이후 `make auto-improve-readiness`를 다시 실행해 queue를 재판정한다.
- readiness report의 `diagnostics.loop_health_summary`는 최신 `ops/reports/routing-provenance-aggregates/<session-id>.json`이 있을 때만 `audit_rollup.loop_health`를 read-only operator summary로 재사용한다. 이 block은 항상 `gate_effect: none`이며 PASS/WARN, checks, remediations, next_action을 바꾸지 않는다. aggregate가 없거나 usable `loop_health`가 없으면 summary는 `status: missing`으로 남는다.
- `auto_improve_policy.workspace_preparation.mode`는 workspace 준비 전략을 선언한다. 기본값은 `full_copy`이며, `sparse_manifest`는 materialized copied-universe 모델로 실행된다.

- `scope-freeze.json`은 focused test/risk resolution뿐 아니라 `auto_improve_policy.allowed_apply_roots`도 materialize한다. 그래서 promotion gate와 live apply 경로가 같은 write allowlist를 직접 감사할 수 있다.

Release freshness의 canonical Make 기본값은 `ARTIFACT_FRESHNESS_MTIME_SOURCE=embedded_currentness`다. `filesystem`은 local hygiene 진단, `zip_info`와 `--zip-metadata ...`는 ZIP/export 전송 검증용 override로 쓴다.
- `make make-target-inventory`는 Makefile의 phony target과 실제 target 정의를 schema-backed `tmp/make-target-inventory.json`으로 산출한다. review/report appendix는 사람이 만든 target 목록 대신 이 artifact를 읽어 absent/present 판단을 고정한다.
- `make outcome-metrics`는 `ops/reports/outcome-metrics.json`을 수동 갱신한다. `make supply-chain-provenance`는 `pyproject.toml`, `requirements.txt`, `requirements-dev.txt`, `uv.lock`, release manifest surface를 읽어 `ops/reports/supply-chain-provenance.json`을 만든다.
- `supply-chain-provenance.json`은 dependency/release provenance용 repo-native audit artifact다. 여기에는 `ci_install_proof`, `lock_evidence`, `provenance_notes`가 들어가며, CI install proof는 `.github/workflows/ci.yml`의 실제 install command surface에서 구조화해 기록한다. `uv.lock` dependency edge는 graph evidence로 파싱하지만, `actions/setup-python`의 cache input이나 `cache-dependency-path`는 cache key evidence일 뿐 install proof로 보지 않는다.
- `make sbom-export-mapping`은 `ops/reports/sbom-export-mapping.json`을 생성한다. 이 report는 `supply-chain-provenance.json`, release manifest, public export surface를 결합해 dependency evidence coverage와 release/public parity를 audit-only로 보여주고, locked dependency edge count도 함께 요약한다.
- `make sbom-readiness-check`는 `ops/reports/sbom-readiness-gate-report.json`을 만든다. `make cyclonedx-sbom`은 `ops/reports/cyclonedx-bom.json`을 생성하며, component/dependency graph는 가능한 경우 `uv.lock` package edge를 사용한다. 이 BOM은 표준 top-level shape를 깨지 않도록 repo-native artifact envelope/currentness를 `metadata.properties`의 `urn:openai:artifact-envelope`에 embedded JSON으로 남기고, artifact freshness가 그 값을 canonical currentness surface로 읽는다.
- `make security-advisories`는 repo-native advisory source를 `ops/reports/security-advisories.json`으로 정규화한다. `make supply-chain-artifact-model`은 CycloneDX, SPDX, OpenVEX, in-toto/SLSA, Sigstore verification metadata가 공통으로 읽는 canonical graph와 cross-artifact ref를 `ops/reports/supply-chain-artifact-model.json`으로 고정한다.
- `make openvex-draft`는 `ops/reports/openvex-draft.json`을 생성한다. advisory input이 없으면 `statements: []`, `vulnerability_source: not_scanned`인 draft shell을 남기고, advisory input이 있으면 justification/action rule을 적용한 statement를 authoring한다. 이 artifact는 repo-local OpenVEX schema를 유지하면서 top-level canonical artifact envelope/currentness metadata도 함께 기록한다.
- `make spdx-sbom`은 `ops/reports/spdx-sbom.json`, `make in-toto-statement`는 `ops/reports/in-toto-statement.json`, `make sigstore-bundle`은 `ops/reports/sigstore-bundle-verification.json`, `make supply-chain-benchmark`는 `ops/reports/supply-chain-benchmark.json`을 생성한다.
- `make openvex-draft-cached`는 security advisories, shared artifact model, CycloneDX, SPDX, OpenVEX, in-toto/SLSA, Sigstore verification metadata를 한 프로세스에서 재사용하며 만드는 canonical fast path다.
- `make generated-artifact-index`는 `ops/reports/generated-artifact-index.json`을 생성해 `ops/reports`, `ops/operator`, `external-reports`, `runs`의 current/archive advisory rule과 archive candidate를 schema-backed로 기록한다. 이 report도 먼저 `tmp/generated-artifact-index.candidate.json`으로 생성된 뒤 검증 promote된다. archive candidate의 실제 이동 증거는 별도 `tmp/archive-execution-manifest.json`이 dry-run/applied/deferred/rollback mode, before/after digest, defer evidence, rollback path, operator confirmation과 함께 남긴다. `make archive-execution-manifest-apply`, `make archive-execution-manifest-defer`, `make archive-execution-manifest-rollback`은 operator lifecycle을 명시적으로 닫는 UX이고, `runs/`의 canonical lifecycle은 physical move가 아니라 `promotion-report.history.status`다.
- `make artifact-freshness-check`는 `tmp/artifact-freshness-report-check.json`만 생성하고 canonical `ops/reports/artifact-freshness-report.json`으로 promote하지 않으며, root `pytest_*` 계열 ephemeral artifact가 남아 있으면 실패한다. canonical freshness report 갱신은 `make artifact-freshness`만 담당한다. 같은 report는 generated JSON artifact의 `$schema`, `generated_at`, artifact envelope, currentness 상태도 transitional attention signal로 집계한다.
- `make manual-mutate-defect-registry`는 `tools/manual_mutate_*.py`를 defect registry 입력으로 승격해 defect class, canonical fix status, regression status를 `ops/reports/manual-mutate-defect-registry.json`에 남기며, 이 report도 canonical generated JSON envelope/currentness contract를 따른다.
- `make supply-chain-check`와 alias인 `make provenance-check`는 generator 결과를 strict gate 후보로 평가해 `ops/reports/supply-chain-gate-report.json`을 만든다. required input missing, parser error, malformed `uv.lock`, CI install proof drift는 fail이다.
- `make release-provenance-clean`과 `make release-sbom-clean`은 optional strict target이다. 기본 `release-check`, `release-clean`, promotion gate surface는 그대로 두고 별도 clean path로만 연결한다.
- `ops/raw-registry.json`은 timestamp 없이, canonical registry의 구조화된 내용과 current inventory에서 파생한 `path_aliases` / `content_sha256`를 함께 반영하는 deterministic export다.
- `ops/manifest.json`은 timestamp 없이, output 자신, editor/cache 디렉터리, packaging 산출물, `ops/reports/`, generated JSON output, binary external reports, `review/` patch workspace를 제외한 deterministic inventory만 담는다.
- repo artifact writer CLI의 `--out`이 상대경로면 `--vault` 기준으로 resolve되고, 대상 parent directory를 공통 helper로 일관되게 생성한 뒤 write한다. 상대 `--out`의 Windows separator는 repo-relative POSIX path로 정규화해 writer output family가 platform path alias로 갈라지지 않게 한다. schema-backed JSON artifact writer는 `ops/scripts/artifact_io_runtime.py`의 `read_json_object`, `load_optional_json_object`, `write_schema_validated_json`, `promote_schema_validated_json`를 canonical helper로 사용하고, legacy per-family helper는 이 계층으로 위임한다. CLI에서는 `ops.scripts.canonical_artifact_promote`가 candidate JSON의 schema, `artifact_kind`, `producer`를 확인한 뒤 canonical path로 승격한다.
- user-export, archive, raw-normalization, curation/scaffold writer는 permissive output path가 허용되는 별도 분류이며, `ops/scripts/script_output_surfaces.py`가 canonical artifact envelope/currentness metadata를 갖춘 `ops/script-output-surfaces.json` registry를 생성한다. 이 registry도 `tmp/script-output-surfaces.candidate.json`에 먼저 쓴 뒤 schema/artifact_kind/producer 검증을 통과하면 promote된다. `tests/test_writer_output_paths.py`는 이 registry를 live AST inventory와 비교해 repo artifact writer의 `resolve_output_path` 사용과 새 output-option writer의 미분류 drift를 정적 테스트로 막는다.
- `ops/`와 `ops/scripts/`는 Python package로 잡혀 있고, 기본 repo-local 실행 경로는 `.venv/bin/python -m ops.scripts...`다. editable install 이후 canonical installed CLI는 `pyproject.toml`의 `[project.scripts]`가 제공하는 `llm-wiki-*` console script다.
- direct-script compatibility가 필요한 entrypoint는 `ops/script-output-surfaces.json`의 `direct_fallback_eligible` 필드로 AST 기반 생성형 registry가 관리한다. 이 목록은 canonical CLI가 아니라 repo-local compatibility를 위한 transitional wrapper surface다. `try/except ImportError`로 import context를 추측하지 않고, `__package__ in (None, "")` branch로만 fallback을 명시한다. direct 실행 branch는 repo root를 `sys.path`에 추가한 뒤 `ops.scripts...` absolute package import를 사용하고, package 실행 branch는 normal relative import를 사용한다. dependency availability check처럼 진짜 import 실패를 계약으로 다루는 경우만 `ImportError`를 잡을 수 있다.
- 안정 import surface는 `ops/script-module-surfaces.json`에 명시하고, 해당 모듈의 literal `__all__`과 같은 목록을 유지한다. 이 contract는 CLI wrapper, compatibility facade, internal facade를 분리해서 direct-script 실행 표면과 import 표면의 drift를 public test에서 잡는다.
- Runtime module을 직접 entrypoint로 유지하지 않는 경우에는 `executor.py`처럼 얇은 CLI wrapper를 두고, runtime module은 package-relative import만 사용한다.

## Runtime 분리 현황

이미 분리된 부분:
- `run_mechanism_experiment_runtime.py`는 orchestration-only wrapper이고, scaffold/capture/workspace/promotion/ledger helper는 `mechanism_run_*_runtime.py` 계열 모듈로 분리되어 있다.
- `run_auto_improve_session()`은 session loop를 유지하되 refresh/select, route/scaffold, execute/evaluate, persist phase helper와 dataclass result로 경계를 드러낸다.
- auto-improve session rollup, proposal queue, execution outcome handling은 각각 `auto_improve_session_runtime.py`, `auto_improve_queue_runtime.py`, `auto_improve_outcome_runtime.py`로 분리되어 있다.
- canonical auto-improve session evidence는 새 artifact family를 만들지 않고 `ops/reports/auto-improve-sessions/<session-id>.json`에 모은다. 각 `iterations[]`는 proposal/scope/routing/executor/run/promotion artifact ref와 normalized `decision`/`outcome`/`failure_taxonomy`/`quarantined`/`primary_targets`를 담고, top-level `learning_summary`가 attempt/rework/rollback/defect-escape/session-context evidence를 요약한다.
- `make generated-artifact-index`가 만드는 `ops/reports/generated-artifact-index.json`은 advisory current/archive rule 외에 `now`, `next`, `why_blocked` operator digest와 item-level `supersedes`, `superseded_by`, `decision_relevance`를 함께 기록한다.
- `promotion_gate_page_runtime.py`와 `promotion_gate_mechanism_runtime.py`는 state 계산, rule evaluator, report assembly helper 경계를 갖고, promotion rule의 order/artifact dependency/reducer/severity/summary template metadata는 `promotion_policy.rule_registry`와 `promotion_policy.rule_metadata`에서 schema-backed contract로 관리한다. `policy_runtime.validate_policy_contract()`는 safety invariant와 registry reference validation을 분리한다.
- `mechanism_review_runtime.py`의 history loading과 candidate emission/calibration은 `mechanism_review_history_runtime.py`, `mechanism_review_candidate_runtime.py`로 분리되어 있다.
- `mechanism_run_scaffold_templates_runtime.py`는 run starter template payload generation을 담당한다.
- `mechanism_run_scaffold_resolution_runtime.py`는 command parsing/executable resolution, proposal input resolution, target/test normalization을 typed helper로 담당한다.
- `wiki_lint_registry_runtime.py`는 얇은 registry lint orchestration façade로 남고, pass support/alignment/review-candidate 계산은 `registry_pass_support_runtime.py`, `registry_alignment_passes_runtime.py`, `registry_review_candidate_passes_runtime.py`로 분리되어 있다.
- `filesystem_runtime.plan_manifest_apply_transaction()`은 explicit canary 경로에서 live write 없이 `shadow-apply-report.json`만 남긴다. live 경로는 shadow plan 뒤 `filesystem_runtime.rehearse_manifest_apply_rollback()`으로 가상 live root에 apply/rollback을 검증하고, `rollback-rehearsal-report.json`이 `pass`일 때만 `filesystem_runtime.apply_manifest_transaction()`을 호출한다. `apply_manifest_transaction()`은 resolved containment와 symlink segment reject를 다시 수행하고, 실패 시 기존 rollback guard를 유지한다. `atomic_multi_write()`는 canonical duplicate target을 hard fail한다.

여전히 미완인 부분:
- `mechanism_run_scaffold_runtime.py`에는 starter extras copy와 scaffold/load orchestration이 아직 함께 남아 있다.
- `auto_improve_runtime.py`는 phase helper가 생겼지만 helper가 같은 모듈에 있어 파일 단위 리뷰 비용은 아직 크다.
- promotion decision registry는 policy 쪽 additive enum만으로는 부족하다. 새 executable decision으로 확장하려면 promotion report schema, trend reducer, finalize/ledger 소비자까지 같은 registry를 읽어야 한다.
- explicit canary apply gate, shadow apply report, live apply rollback rehearsal은 운영 표면에 올라왔다.
- repo-level 공급망 provenance generator는 `supply_chain_provenance.py`, strict evaluator는 `supply_chain_gate_runtime.py`, release/public dependency evidence mapping은 `sbom_export_mapping.py`, shared graph layer는 `supply_chain_artifact_model.py`, CycloneDX emitter는 `cyclonedx_sbom.py`, SPDX emitter는 `spdx_sbom.py`, OpenVEX authoring은 `openvex_draft.py`, in-toto/SLSA emitter는 `in_toto_statement.py`, Sigstore verification metadata는 `sigstore_bundle.py`로 운영 표면에 올라왔다. generator와 gate를 분리해 audit evidence와 strict policy 판정을 분리한 것이 현재 canonical 구조다. CycloneDX BOM은 표준 HTTP `$schema` URI를 유지하지만, `schema_constants_runtime.py`의 local alias가 vendored `ops/schemas/cyclonedx-1.6.schema.json`으로 해석하므로 validation은 offline으로 닫힌다. release closeout에는 기본 편입하지 않고 `RELEASE_CLOSEOUT_PROFILE=provenance|sbom`으로 opt-in한다.
- 남은 후속 과제는 content-addressed run-local provenance snapshot과 default release hard gate 승격 여부 판단이다. PyPI Trusted Publishing + provenance attestation wiring은 `.github/workflows/release.yml`에 반영됐다.

다음 개선 기준:
- 1순위는 promotion decision registry를 downstream schema/reducer/finalize 경로까지 맞추는 것이다.
- 2순위는 `mechanism_run_scaffold_runtime.py`의 starter extras copy와 scaffold/load orchestration을 더 나눌 가치가 있는지 실제 변경 압력을 기준으로 판단하는 것이다.
- `run_auto_improve_session()`의 phase helper는 지금처럼 session loop 안에 남겨도 되며, 독립 fault injection이나 외부 재사용 요구가 생길 때 파일 단위 분리를 검토한다.

## 운영 원칙
- score를 바꾸는 정책 변경은 `system/system-log.md`에 남긴다.
- lint / eval 기준은 갑자기 느슨하게 만들지 않는다.
- `system/system-log.md`는 lint 대상에서 제외하지 않는다. 대신 log append를 validation 전에 수행해 final lint/eval loop가 중복 실행되지 않게 한다.
- source fidelity를 깨는 자동화는 금지한다. 다만 `raw/**/*.md`에는 metadata backfill과 capture noise 제거 수준의 minimal post-ingest normalization만 별도 contract로 허용한다.
- 읽기 전용 gate와 generated artifact 갱신 경로를 섞지 않는다.
- historical promotion backfill은 기본 운영 범위에 포함하지 않는다.
